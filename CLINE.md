# CLINE.md — GENPRO Revit Plugins

Инструкция для Cline при работе в этом репозитории.

Цель: воспроизводить поведение Claude Code в части ведения и сохранения сессии, с обязательной фиксацией контекста, изменений, planning-состояния и итогов в Obsidian.

---

## 1. Базовый режим работы

Перед началом любых изменений:

1. Прочитай `CLAUDE.md`.
2. Прочитай `README.md` и `.github/copilot-instructions.md`.
3. Прочитай `DG_OBSIDIAN/00-home/index.md`.
4. Прочитай `DG_OBSIDIAN/текущие приоритеты.md`.
5. Если задача относится к конкретному модулю — найди и прочитай связанную заметку в `DG_OBSIDIAN/knowledge/`.

После этого кратко зафиксируй для себя:
- цель текущей сессии;
- какие файлы, вероятно, будут изменены;
- какие есть архитектурные ограничения;
- какие артефакты нужно будет сохранить в Obsidian по завершении.

---

## 2. Ветки и режим работы по GSD

Правила:
- для каждой активной фазы GSD используй отдельную feature-ветку соответствующего релиза;
- не смешивай изменения разных релизов в одной ветке;
- если задача относится к конкретной фазе или релизу GSD, сначала убедись, что работа ведётся в правильной ветке;
- если ветка не соответствует релизу, сообщи об этом пользователю перед началом изменений.

---


## 3. Сборка и коммиты

После обычных изменений:
- не выполняй сборку проекта автоматически;
- не создавай коммиты автоматически;
- не делай промежуточные коммиты;
- не предлагай коммит после каждого изменения.

Коммиты разрешены только по явной команде пользователя:

`Сохрани сессию`

Сборку проекта выполняй только если пользователь прямо попросил:
- собрать проект;
- проверить сборку;
- прогнать валидацию;
- или это явно требуется отдельной задачей.

---


## 4. Учёт текущей сессии

С самого начала сессии веди внутренний список:

- файлов, созданных в этой сессии;
- файлов, изменённых в этой сессии;
- решений, принятых в этой сессии;
- багов, найденных или исправленных в этой сессии;
- изменений planning/GSD;
- заметок, которые нужно будет создать в Obsidian.

Важно:
- учитывай только изменения текущей сессии;
- не включай в итог старые незакоммиченные изменения из предыдущих сессий;
- одна сессия = один отдельный session record;
- одна сессия = один отдельный коммит, если были изменения файлов и пользователь дал команду `Сохрани сессию`.

---

## 7. Запрет на запись session data в CLINE.md

`CLINE.md` — это постоянная инструкция агента, а не журнал сессий.

При выполнении команды `Сохрани сессию` запрещено:
- перезаписывать `CLINE.md`;
- дописывать в `CLINE.md` информацию о текущей сессии;
- использовать `CLINE.md` как место для хранения результатов, логов, handoff или session summary.

Информация о сессии должна записываться только в предназначенные для этого артефакты:

- `DG_OBSIDIAN/sessions/` — итоговая заметка сессии;
- `DG_OBSIDIAN/текущие приоритеты.md` — обновление текущих приоритетов;
- `DG_OBSIDIAN/knowledge/decisions/` — заметки о решениях;
- `DG_OBSIDIAN/knowledge/debugging/` — заметки о багах и расследованиях;
- `DG_OBSIDIAN/00-home/index.md` — обновление индекса при появлении новых заметок;
- `.planning/STATE.md` — только если текущая сессия изменила состояние исполнения GSD;
- `.planning/ROADMAP.md`, `.planning/REQUIREMENTS.md`, `.planning/PROJECT.md`, `.planning/release-map.json` — только через workflow `sync-docs`, если это требуется задачей.

Изменять `CLINE.md` можно только в одном случае:
- пользователь явно просит изменить саму инструкцию агента.

---

## 8. Триггер "Сохрани сессию"

Если пользователь пишет:
- `Сохрани сессию`
- `сохрани сессию`
- `save session`
- или формулирует явный запрос на завершение и фиксацию сессии,

выполни весь post-session workflow без дополнительных уточнений.

### 8.1. Создай session note

Создай новую заметку в:
`DG_OBSIDIAN/sessions/`

Имя файла должно содержать дату сессии.

Используй строго этот шаблон:

```md
сессия: <краткое описание работы за сессию в одном предложении>

## Информация о сессии
- Модель: <текущая LLM модель>
- Дата: <YYYY-MM-DD>
- Изменено файлов: <N>

## Изменённые файлы
- <список файлов, изменённых в этой сессии>

## Результаты
- <ключевое изменение 1>
- <ключевое изменение 2>
- <ключевое изменение 3>
```

Правила заполнения:
- описание сессии — одно краткое предложение;
- дата — дата последних изменений;
- список файлов — только файлы текущей сессии;
- результаты — только реально выполненные изменения.

### 8.2. Обнови приоритеты

Обнови файл:
`DG_OBSIDIAN/текущие приоритеты.md`

Отрази:
- что завершено;
- что осталось следующим шагом;
- есть ли блокеры;
- что перенесено на следующую сессию.

### 8.3. Создай knowledge note при необходимости

Если в сессии было принято решение:
- создай заметку в
  `DG_OBSIDIAN/knowledge/decisions/`

Если в сессии был исследован, локализован или исправлен баг:
- создай заметку в
  `DG_OBSIDIAN/knowledge/debugging/`

### 8.4. Обнови индекс

Если в Obsidian созданы новые заметки, обнови:
`DG_OBSIDIAN/00-home/index.md`

Добавь ссылки на новые материалы, если это требуется текущей структурой хранилища.

### 8.5. Обнови planning/GSD только по делу

Если текущая сессия реально изменила planning-состояние или потребовала синхронизации:
- обнови `.planning/STATE.md`, если нужно зафиксировать текущую фазу, прогресс или decisions log;

### 8.6. Создай коммит только по текущей сессии

Если в текущей сессии были изменены файлы:
- подготовь коммит только из файлов, созданных или изменённых именно в этой сессии;
- не включай изменения, оставшиеся от прошлых сессий;
- не смешивай несколько сессий в один коммит;
- создавай коммит только после явной команды `Сохрани сессию`.

Если файловых изменений в этой сессии нет:
- коммит не создавай.

## 9. Формат session handoff

Если нужно передать работу другому агенту, формируй блок:

```text
SESSION_HANDOFF
Цель: <что делали>
Контекст: <почему это делается>
Ветка: <текущая ветка, например feature\phase02>
Сделано: <что уже завершено>
Изменённые файлы: <список>
Открытые вопросы: <что осталось>
Решения: <принятые решения>
Obsidian actions: <что нужно записать в sessions / decisions / debugging>
Planning/GSD state: <нужен ли resync, менялся ли STATE.md>
```

Этот handoff должен отражать только текущее состояние работы и быть достаточным для продолжения сессии в Claude Code без чтения полного диалога.

---

## 10. Приоритет поведения

Если возникает конфликт между скоростью и полнотой процесса, соблюдай порядок приоритетов:

1. Не нарушать архитектурные и кодовые ограничения репозитория.
2. Работать в правильной feature-ветке релиза GSD.
3. Не выполнять лишнюю сборку и не делать коммиты без команды `Сохрани сессию`.
4. Не записывать session data в `CLINE.md`.
5. Корректно выполнить изменения по задаче.
6. Корректно зафиксировать результаты в Obsidian.
7. Корректно обновить planning/GSD-файлы, если это требуется задачей.
8. Корректно ограничить коммит только файлами текущей сессии.
9. Сформировать session handoff, если работа будет продолжена другим агентом.

---

## 11. Ожидаемое поведение Cline

Cline должен работать не как чат с произвольной памятью, а как агент с дисциплиной Claude Code:

- начинать с чтения проектного и Obsidian-контекста;
- соблюдать repository rules;
- работать в отдельной feature-ветке релиза GSD;
- отслеживать scope текущей сессии;
- не выполнять сборку автоматически после изменений;
- не делать коммиты автоматически после изменений;
- не записывать информацию о сессии в `CLINE.md`;
- по команде `Сохрани сессию` сохранять результаты в Obsidian и связанные planning/GSD-файлы;
- оставлять после себя воспроизводимый набор артефактов:
  - session note;
  - updated priorities;
  - decision/debugging notes при необходимости;
  - обновлённый index;
  - planning/GSD updates при необходимости;
  - отдельный session commit;
  - handoff для продолжения работы.

Работай так, чтобы любую сессию можно было без потерь продолжить в Claude Code.
