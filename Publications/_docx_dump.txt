[2][-] 3 slide. Hook
[3][-] How much design intent survives the next handover?
[4][-] Illustrate how design intent is stripping away through the architectural design development aligned to the phases of RIBA plan
[5][-] “From Concept Design to Maintenance – stripping away design intent”
[6][-] Currently, architectural design largely adheres to handover principles throughout the development process. Consequently, with each transfer of design outputs among stakeholders, the design intent and rules established before decisions are lost. Each transition causes a partial loss of intent because only final states are exchanged as detached files, not the underlying rules and constraints that defined them
[7][-] The intent in order to be sustainable and constant throughout the design lifecycle should be encoded. The encoded intent is used to be called Design Knowledge.

[30][-] 9 slide. The Knowledge Gap.
[31][a4] BIM manages information — but design knowledge stays fragmented across files, platforms and disciplines.
[32][-] While Building Information Modelling (BIM) has advanced to a high degree of automation in project information management, design knowledge remains fragmented across various domain-specific datasets that are continuously reconstructed across tools and project phases, resulting in redundancy, loss of original intent, and limited traceability
[33][a4] Parametric tools (Grasshopper / Dynamo) silo that logic as opaque,​non-interoperable node graphs that die at export — the rules are​replicated per platform, never shared.​

To address the fragmentation of architectural knowledge inherent in conventional design workflows, a shift is required from detailed representations to abstract, object-level knowledge, thereby establishing a common semantic tier that connects domain-specific parts of the model without duplicating information.

[42][-] 12 slide. The 3 pillars of Design Grammars
[43][-] From the goal above, main characteristics of proposed system are stated as three pillars: Explicit Machine-Readable, Generative or Emergent nature, Cross-platform or tool independent. Based on these pillars, the technologic solution core of the system is Ontology-driven Knowledge Graph.

[45][-] 13 slide. Knowledge Graph
[46][-] A knowledge graph is an organized representation of information that models real-world entities and the relationships between them. It maps data visually as a network of connected labeled entities with properties as nodes and edges, so the data keeps its meaning and context.
[47][-] The Graph is used to encode Design Grammars from diverse sources of Knowledge, keeping multilayer concept relationships and reusing it.
[48][-] 
[49][-] 14 slide. Ontology. Definition
[50][-] Ontology is a structured, machine-readable framework that organizes knowledge within a specific domain. It consists of:
[51][-] Classes: The concepts or types of things (e.g., Employee, Manager).
[52][-] Relationships (Object Properties): How these classes connect to one another (e.g., Manages, Works For).
[53][-] Properties (Data Properties): Describing classes with specific value (literal)
[54][-] Instances: The actual, concrete entities (e.g., the person "John Doe").

[59][-] 16 slide. DG System Framework.
[60][-] Proposed DG system framework represents a Core-Periphery data model and consists of 5 main layers sorted from the outer layer to the core. 
[61][-] This 5-layer framework structures how architectural design intent is captured, processed, and validated. It translates human input through the Communication layer into structured logic via Integration and Orchestration (AI/APIs). That logic is evaluated in the Analysis layer and stored persistently in the Data layer (Graph DB). Together, this stack turns abstract knowledge into executable rules, ensuring design intent is actively validated against the BIM model rather than lost in file exchanges.
