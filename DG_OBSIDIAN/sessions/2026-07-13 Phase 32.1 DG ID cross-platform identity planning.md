# Session: Phase 32.1 — Cross-Platform Identity and Mapping (DG ID) planning

## Trigger

User directive via `/gsd-plan-phase`: insert a new phase into v9.0 covering cross-platform object identity — the same conceptual design object (e.g. a parametric wall defined in Grasshopper and the BIM wall generated from it in Revit) must resolve to one identity within a Design State, so properties computed on one platform (worked example: Ladybug-derived insulation on a facade panel, calculated in Grasshopper) become readable from every other platform's bound representation (the Revit panel). User explicitly required the state of the art to be researched via Perplexity MCP, naming two targets: Rhino.Inside.Revit's GH↔Revit object identification, and Speckle's cross-platform object identification.

2. **Perplexity SOTA sweep** (orchestrator-level, sequential calls — see [[debugging/perplexity-mcp-parallel-call-crashes-browser-profile|debugging note]]):
   - **Rhino.Inside.Revit Element Tracking**: only *Add* components track; the component *output* stores the binding to created Revit elements, persisted across file close/reopen; three modes — Disabled / Enabled:Replace / Enabled:Update (default: modify via Revit API where possible, else recreate); management actions Highlight/Unpin/Delete/**Release** (Release forgets the binding → duplicates on rerun). Revit `ElementId` is session/project-local and unstable; `UniqueId` (episode GUID + element-id tail) is the durable cross-session identity.
   - **Speckle two-layer identity**: `id` = content hash of serialized state (changes with content); `applicationId` = stable source-app identifier, the long-term join key across commits/versions. Revit connector sets `applicationId` = Revit `UniqueId`; matching `applicationId` on receive triggers update-in-place, else delete+recreate; no `applicationId` → no update-in-place. **Known gap: Grasshopper-originated objects generally lack a stable `applicationId`** — exactly the hole DG ID fills.
   - **Landscape**: IFC `GlobalId` = 22-char compressed GUID, must persist for `IfcRoot` object lifetime. BHoM's Revit adapter stores identity as a `RevitIdentifiers` *fragment* (PersistentId=UniqueId, ElementId, category/family/host) separate from geometry — matches on push via the fragment. Best-practice pattern: canonical pipeline ID (deterministic minting from immutable inputs) + a registry recording first-seen source→target mappings (safer for reorder/split/merge); derived objects get explicit parent lineage rather than invented identity.
3. **Research/pattern-mapping subagents** — `gsd-phase-researcher` (RESEARCH.md) and `gsd-pattern-mapper` (PATTERNS.md) run against the CONTEXT.md + Perplexity seed findings. ## Key decisions

See [[decisions/Phase 32.1 DG ID cross-platform identity design|decision note]] for the full rationale — summarized:
- `dgId` wraps `cgId` (doesn't replace it), deterministic SHA-256-hex16 hash with `project` folded in
- Native platform ids (GH GUID, Revit UniqueId, IFC GlobalId, Speckle applicationId) are *representations* bound to the dgId in a registry — never the identity itself
- Registry uses a dedicated `Representation` node + `HAS_REPRESENTATION` edge (not flat properties)
- Shared cross-platform properties live in `SharedProperty` records with platform/connector/timestamp provenance, not direct overwrites
- Rename default: re-mint (not preserve) — an escape hatch for preserve-on-rename is documented as a future refinement, not built in this phase
- `Representation`/`SharedProperty` nodes live under `graph:'Computgraph'` (not a new partition)
- Revit `nativeIdKind` in bindings defaults to `UniqueId`, never `ElementId` (Rhino.Inside.Revit lesson)
- Real Revit connector is explicitly out of scope — this phase proves the contract via a simulated second-platform consumer only
