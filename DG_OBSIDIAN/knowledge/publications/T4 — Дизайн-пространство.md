# T4 — Дизайн-пространство: анализ, генерация и предсказание

## Полное название

**Design Space Analysis, Generation and Prediction in Constraint-Driven Parametric Design Workflows**

## Аннотация (краткая)

Формализует дизайн-пространство как коллекцию троек (вектор параметров, профиль соответствия, вектор метрик), накопленных за сессию. Вводит четвёртый граф-слой DesignSpaceGraph, язык операторских метрик MetricSpec, ретроспективное вычисление метрик, генерацию состояний через LLM-предложения и Latin hypercube sampling. Кейс-стади: продолжение T3 (mixed-use башня), 3 оператора-метрики, 5 сгенерированных кандидатов, улучшение эффективности на 5.7%.

## Ключевые вклады (не пересекаются с T1/T2/T3)

1. **Формальное определение дизайн-пространства**: Σ = коллекция DesignSpacePoint (StateId, вектор параметров, профиль соответствия, вектор метрик)
2. **DesignSpaceGraph** (4-й граф-слой): MetricSpec, DesignSpacePoint, GeneratedCandidate
3. **Отношения DesignSpaceGraph**: HAS_METRIC_VALUE (с числовым value), LINKED_RUN (→ ValidationRun), GENERATED_FROM (метод: llm|grid|lhs)
4. **MetricSpec**: metricId, name, expression (арифметика по ParameterIds), direction (maximize/minimize/target), targetValue, unit
5. **Язык операторских метрик**: NL-промпт → LLM → выражение → валидация → MERGE MetricSpec в Neo4j
6. **Ретроспективный сервис**: POST /design-space/compute/{project}, идемпотентный, async для >200 прогонов
7. **LLM-assisted suggestion**: новый n8n-эндпоинт, chain-of-thought промптинг (temp=0.3), JSON-вектор параметров
8. **Latin hypercube sampling**: стратифицированная выборка по доменным границам из истории, аналитический пре-фильтр SWRL
9. **DESIGN SPACE-компонент GH**: 6 входов + 5 выходов (MetricVector, SpacePoints, NearestCompliant, Candidates, Status)
10. **Parallel coordinates UI**: D3.js v7, ось-ориентация drag&drop, localStorage-персистентность фильтров

## Ключевые концепции

- **Дизайн-пространство** = множество дизайн-решений, согласованных с дизайн-состояниями (DesignStateSnapshot)
- **Анализ дизайн-пространства** = сравнение состояний по операторским метрикам, поиск Парето-оптимальных точек
- **Генерация** = заполнение пространства кандидатами через LLM + LHS, с пре-фильтрацией по SWRL
- **Предсказание** = NearestCompliant — ближайшее соответствующее историческое состояние к целевому вектору метрик
- **Оператор DG-системы** = тот, кто задаёт метрики и желаемые целевые значения через NL-промпт

## Кейс-стади

- Продолжение T3: mixed-use башня, 12 правил, 9 исторических прогонов
- Три операторских метрики: Floor Efficiency (DS_M_EFFICIENCY), Compactness (DS_M_COMPACTNESS), Solar Exposure Proxy (DS_M_SOLAR)
- LLM сгенерировал Gen-01..03; LHS — Gen-04..05
- Gen-02 подтверждён полностью соответствующим: +5.7% к эффективности vs лучший ручной прогон
