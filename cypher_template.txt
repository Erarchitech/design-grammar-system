DG CYPHER TEMPLATE FOR LLM (v4)

SCHEMA VERSION: v4.0

TASK
Convert natural-language design rules into Neo4j Cypher MERGE + SET
statements that populate OntoGraph (ontology terms) and Metagraph
(rules, atoms, variables, literals, builtins).

OUTPUT RULES
- Output Cypher only — no JSON, no markdown fences, no commentary.
- Use only MERGE and SET.
- Allowed node labels: Class, DatatypeProperty, ObjectProperty, Builtin,
  Rule, Atom, Var, Literal.
- Allowed relationship types: HAS_BODY, HAS_HEAD, REFERS_TO, ARG.
- Every persisted node MUST include `project` and `graph` properties.
- graph values: 'OntoGraph' for Class / DatatypeProperty / ObjectProperty;
  'Metagraph' for Rule / Atom / Builtin / Var / Literal.

GRAPH SCHEMA (from dataset_schema v3)

  Rule   — key: Rule_Id
           props: SWRL (SWRL string), kind (violation|compliance),
                  RuleName (optional), RuleDescription (optional),
                  project, graph='Metagraph'

  Class  — key: iri (ex:<ClassName>)
           props: label, project, graph='OntoGraph'

  DatatypeProperty — key: iri (ex:<propertyName>)
           props: SWRL_label, range (xsd:decimal|xsd:integer|…),
                  project, graph='OntoGraph'

  ObjectProperty — key: iri (ex:<propertyName>)
           props: label, range, project, graph='OntoGraph'

  Builtin — key: iri (swrlb:<builtinName>)
           props: label, project, graph='Metagraph'

  Atom   — key: Atom_Id ({Rule_Id}_A{n} for body, {Rule_Id}_H{n} for head)
           props: type (ClassAtom|DataPropertyAtom|ObjectPropertyAtom|BuiltinAtom),
                  iri, SWRL_label, project, graph='Metagraph'

  Var    — key: name (?<varName>) + project
           props: project, graph='Metagraph'

  Literal — key: lex + datatype
           props: project, graph='Metagraph'

  ~~ DesignState & Run (document-only — NOT emitted by LLM ingest) ~~

  These nodes are written only by VALIDATOR-on-publish, not by LLM rule-ingest.
  The ingest emits OntoGraph + Metagraph nodes only. This subsection documents
  the full schema for reader context.

  DesignState — key: StateId + project
           props: kind (ObjState|ParamState|PropState), project, graph='ValidGraph'

  Run   — key: ValidationRunId + project
           props: ValidStatus (per-ObjState Boolean list, index-matched),
                  SendStatus (single Boolean),
                  project, graph='ValidGraph'
           NOTE: No stored Status text field. Overall pass = AND(ValidStatus),
                 derived at read time.

  Run→DesignState relationship: (:Run)-[:VALIDATES]->(:DesignState)
           Cardinality: many Runs to one DesignState.

  ~~ Identity-registry nodes (document-only — NOT emitted by LLM ingest) ~~

  These nodes are written ONLY by the data-service identity API
  (/identity/mint, /identity/bind, /identity/{dgId}/properties), never by LLM
  rule-ingest. The ingest emits OntoGraph + Metagraph nodes only. This
  subsection documents the registry shape for reader context.

  Representation — key: (nativeId, platform, project)
           graph: 'Computgraph'
           props: nativeId, platform (Grasshopper|Revit|IFC|Speckle),
                  nativeIdKind (InstanceGuid|UniqueId|GlobalId|ApplicationId),
                  connector, boundAt

  SharedProperty — key: (dgId, propertyName, project)
           graph: 'Computgraph'
           props: dgId, propertyName, value, platform, connector, writtenAt

  Entity→Representation   relationship: (:Computgraph entity)-[:HAS_REPRESENTATION]->(:Representation)
  Entity→SharedProperty  relationship: (:Computgraph entity)-[:HAS_SHARED_PROPERTY]->(:SharedProperty)

  Every Computgraph entity node (Object, Procedure, Pattern, Parameter,
  Interface) may carry dgId — a deterministic platform-neutral identity
  (format dg:<16 uppercase hex>, see spec/DG-ID.md), NOT emitted by LLM ingest.
  The template's MERGE blocks must never create these nodes or relationships.

  ~~ Computgraph runtime nodes (document-only — NOT emitted by LLM ingest) ~~

  These nodes are written ONLY by the POST /computgraph/publish endpoint
  (data-service/app.py), not by LLM rule-ingest. This subsection documents the
  Computgraph runtime labels, properties, and relationships for reader context.

  Node labels (all graph:'Computgraph', project):

  Object — key: (cgId, definitionId, project)  [display: objectName]
           optional: classIri (cross-layer link to OntoGraph Class)
           props: objectName, classIri, dgId, source (tagged|recognized),
                  provider, model, confidence, definitionId, fileName, publishedAt

  Behavior — key: (definitionId, project)  [display: none]
           props: definitionId
           NOTE: structural node — no cgId/dgId

  Algorithm — key: (algIndex, definitionId, project)  [display: algorithmName]
           props: algorithmName, algIndex, contextJson

  Procedure — key: (cgId, definitionId, project)  [display: procedureName]
           props: procedureName, procIndex, cgId, dgId

  Pattern — key: (cgId, definitionId, project)  [display: patternName]
           props: patternName, cgId, dgId

  Parameter — key: (cgId, definitionId, project)  [display: parameterName]
           props: parameterName, paramKind (Variable|Constant|Emergent),
                  dataType (Float|Integer|Text|Boolean|Geometry),
                  domainMin, domainMax, domainStep (optional), cgId, dgId

  Interface — key: (cgId, definitionId, project)  [display: interfaceName]
           props: interfaceName, ifaceType (Input|Output), cgId, dgId

  Relationships:

  HAS_BEHAVIOR     (Object→Behavior)
  HAS_ALGORITHM    (Behavior→Algorithm)
  HAS_PROCEDURE    (Algorithm→Procedure)
  HAS_PATTERN      (Procedure→Pattern)
  PATTERN_HOST_TO  (Pattern→Pattern, nesting)
  HAS_PARAMETER    (Procedure→Parameter)
  HAS_INTERFACE    (Procedure→Interface)
  PARAM_LINK       (Parameter→Interface, wire-derived linking)
  REFERS_TO        (Object→Class, cross-layer bridge to OntoGraph)

  Entity→Representation  relationship: (:Computgraph entity)-[:HAS_REPRESENTATION]->(:Representation)
  Entity→SharedProperty  relationship: (:Computgraph entity)-[:HAS_SHARED_PROPERTY]->(:SharedProperty)

SEMANTIC MAPPING (violation pattern)
- "Maximum / at most / no more than X" → body uses swrlb:greaterThan(?val, X)
- "Minimum / at least / no less than X" → body uses swrlb:lessThan(?val, X)
- "Equal to X"               → body uses swrlb:notEqual(?val, X)
- "Between min and max"      → emit TWO separate violation rules (one for min, one for max)
Body checks the condition that *breaks* the rule; head sets the violation flag to true.

RULE SEPARATION PRINCIPLE
- Each distinct constraint MUST be a separate Rule node with its own Atom subgraph.
- NEVER combine multiple constraints (e.g. "at least 2 toilets AND at least 1 kitchen")
  into a single Rule.
- If the prompt contains N constraints, output N separate Rule subgraphs.
- Shared Class / DatatypeProperty / ObjectProperty nodes are reused via MERGE (same iri).
- Each Rule gets its own violation DatatypeProperty.

PROPERTY REUSE ON EDIT
- When editing an existing rule, preserve the same DatatypeProperty iri and SWRL_label
  that the original rule used. Only change Literal lex values for numeric threshold changes.

IDENTIFIERS
- Rule_Id format: R_<DOMAIN>_<PROPERTY>_<LIMIT>_V
                  (e.g. R_URB_HEIGHT_MAX_75_V)
- Body atom ids:  {Rule_Id}_A1, _A2, _A3, …
- Head atom ids:  {Rule_Id}_H1, _H2, …

IRI PREFIXES
- ex:    for domain terms (classes, properties, violation predicates)
- swrlb: for SWRL builtins

CYPHER TEMPLATE BLOCK

// 1. OntoGraph — ontology terms
MERGE (C_Main:Class {iri: 'ex:<CLASS_NAME>'})
SET C_Main.label = '<CLASS_NAME>',
    C_Main.project = '<PROJECT>',
    C_Main.graph = 'OntoGraph'

MERGE (DP_Main:DatatypeProperty {iri: 'ex:<PROPERTY_NAME>'})
SET DP_Main.SWRL_label = '<PROPERTY_NAME>',
    DP_Main.range = '<xsd:decimal|xsd:integer|xsd:boolean>',
    DP_Main.project = '<PROJECT>',
    DP_Main.graph = 'OntoGraph'

MERGE (B_Cmp:Builtin {iri: 'swrlb:<BUILTIN_NAME>'})
SET B_Cmp.label = '<BUILTIN_NAME>',
    B_Cmp.project = '<PROJECT>',
    B_Cmp.graph = 'Metagraph'

MERGE (DP_V:DatatypeProperty {iri: 'ex:<VIOLATION_PROPERTY>'})
SET DP_V.SWRL_label = '<VIOLATION_PROPERTY>',
    DP_V.range = 'xsd:boolean',
    DP_V.project = '<PROJECT>',
    DP_V.graph = 'OntoGraph'

// 2. Metagraph — rule node
MERGE (r:Rule {Rule_Id: '<RULE_ID>'})
SET r.SWRL = '<SWRL_EXPRESSION>',
    r.kind = 'violation',
    r.project = '<PROJECT>',
    r.graph = 'Metagraph'
    // r.RuleName and r.RuleDescription are optional — LLM sets them when the
    // NL rule text supplies a name/description; omit the SET clause otherwise.

// 3. Metagraph — variables, literals
MERGE (vEntity:Var {name: '?<ENTITY_VAR>', project: '<PROJECT>'})
SET vEntity.graph = 'Metagraph'

MERGE (vValue:Var {name: '?<VALUE_VAR>', project: '<PROJECT>'})
SET vValue.graph = 'Metagraph'

MERGE (litX:Literal {lex: '<THRESHOLD>', datatype: '<xsd:decimal|xsd:integer>'})
SET litX.project = '<PROJECT>',
    litX.graph = 'Metagraph'

MERGE (litT:Literal {lex: 'true', datatype: 'xsd:boolean'})
SET litT.project = '<PROJECT>',
    litT.graph = 'Metagraph'

// 4. Metagraph — atom nodes
MERGE (a1:Atom {Atom_Id: '<RULE_ID>_A1'})
SET a1.type = 'ClassAtom',
    a1.iri = 'ex:<CLASS_NAME>',
    a1.SWRL_label = '<CLASS_NAME>',
    a1.project = '<PROJECT>',
    a1.graph = 'Metagraph'

MERGE (a2:Atom {Atom_Id: '<RULE_ID>_A2'})
SET a2.type = 'DataPropertyAtom',
    a2.iri = 'ex:<PROPERTY_NAME>',
    a2.SWRL_label = '<PROPERTY_NAME>',
    a2.project = '<PROJECT>',
    a2.graph = 'Metagraph'

MERGE (a3:Atom {Atom_Id: '<RULE_ID>_A3'})
SET a3.type = 'BuiltinAtom',
    a3.iri = 'swrlb:<BUILTIN_NAME>',
    a3.SWRL_label = '<BUILTIN_NAME>',
    a3.project = '<PROJECT>',
    a3.graph = 'Metagraph'

MERGE (h1:Atom {Atom_Id: '<RULE_ID>_H1'})
SET h1.type = 'DataPropertyAtom',
    h1.iri = 'ex:<VIOLATION_PROPERTY>',
    h1.SWRL_label = '<VIOLATION_PROPERTY>',
    h1.project = '<PROJECT>',
    h1.graph = 'Metagraph'

// 5. Relationships — rule ↔ atoms
MERGE (r)-[:HAS_BODY {`order`: 1}]->(a1)
MERGE (r)-[:HAS_BODY {`order`: 2}]->(a2)
MERGE (r)-[:HAS_BODY {`order`: 3}]->(a3)
MERGE (r)-[:HAS_HEAD {`order`: 1}]->(h1)

// 6. Relationships — atoms → ontology terms (REFERS_TO)
MERGE (a1)-[:REFERS_TO]->(C_Main)
MERGE (a2)-[:REFERS_TO]->(DP_Main)
MERGE (a3)-[:REFERS_TO]->(B_Cmp)
MERGE (h1)-[:REFERS_TO]->(DP_V)

// 7. Relationships — atoms → arguments (ARG)
MERGE (a1)-[:ARG {`pos`: 1}]->(vEntity)
MERGE (a2)-[:ARG {`pos`: 1}]->(vEntity)
MERGE (a2)-[:ARG {`pos`: 2}]->(vValue)
MERGE (a3)-[:ARG {`pos`: 1}]->(vValue)
MERGE (a3)-[:ARG {`pos`: 2}]->(litX)
MERGE (h1)-[:ARG {`pos`: 1}]->(vEntity)
MERGE (h1)-[:ARG {`pos`: 2}]->(litT);

QUALITY CHECKS
- No malformed MERGE syntax.
- Every Atom has exactly one REFERS_TO pointing to a Class, DatatypeProperty, ObjectProperty, or Builtin.
- Every ARG relationship includes `pos`.
- Rule key is Rule_Id (not id).
- Atom key is Atom_Id (not id).
- DatatypeProperty display property is SWRL_label (not label).
- Rule SWRL body property is named SWRL (not text).
- Backtick-quote reserved words in relationship properties: `order`, `pos`.
- Cypher must execute in Neo4j 5.
