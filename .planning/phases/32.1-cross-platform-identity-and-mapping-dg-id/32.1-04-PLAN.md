New symbols in `data-service/dg_identity.py`:
- pydantic model `SharedPropertyWriteRequest` (`property_name`, `value`, `platform`, `connector`, `project`)
- helpers `write_shared_property(session, dg_id, property_name, value, platform, connector, project) -> dict` and `read_shared_property(session, dg_id, property_name, project) -> dict | None`

- Routes `GET /identity/{dg_id}/properties` (query `project`, optional `property_name`) and `POST /identity/{dg_id}/properties` (body `SharedPropertyWriteRequest`)

All symbols above are NET-NEW in Phase 32.1. This plan extends the module and routes created in Plan 03.

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-32.1-04a | Tampering | Cypher injection via property name/value | high | mitigate | Parameterized Cypher only; the property name/value/platform are bound parameters, never interpolated — grep gate enforced, same as Plan 03. |
| T-32.1-04b | Spoofing | SharedProperty for a non-existent dgId | medium | mitigate | `write_shared_property` verifies the entity `{dgId, project}` exists and raises `DGID_NOT_FOUND` before any MERGE — a forged/unminted dgId cannot accrete orphan properties. |
| T-32.1-04c | Repudiation | which platform computed a value | medium | mitigate | Every write records `platform`, `connector`, and `writtenAt`; the read returns provenance, so a value's origin is always attributable (the DGID-04 provenance requirement). |
| T-32.1-04d | Information Disclosure | cross-project property leakage | high | mitigate | Read/write MERGE/MATCH keys include `project`; property reads are project-scoped, tested. |

No package-manager installs in this plan — no T-32.1-SC legitimacy gate required.
