<domain>
## Phase Boundary

Design Grammars becomes a cross-platform system: the same conceptual design object exists as different native representations on different platforms (a parametric wall defined in Grasshopper; the BIM wall generated from it in Revit; its Speckle-published copy). This phase delivers the **identity spine** that makes those representations one object:

- a durable, platform-neutral **DG ID** minted for every design object extracted into the Computgraph,
- an **identity registry** in Neo4j binding each DG ID to its per-platform native representations (GH instance GUID, Revit UniqueId/ElementId, IFC GlobalId, Speckle applicationId),
- **shared cross-platform properties** attached to the DG ID — computed on one platform, readable from all others,
- a **data-service identity API** (mint / resolve / bind / property read-write) that all current and future DG connectors speak.

In scope: the spec (`spec/DG-ID.md`), DG.Core identity model + deterministic minting, `cgContextJson` envelope carrying `dgId`, Neo4j registry shape, data-service API, a simulated-second-platform proof of the property-flow contract, and the ADR.

Out of scope: the actual Revit connector (future milestone — this phase proves the contract against a simulated Revit-side consumer); any change to Phase 32's five existing plans (identity layers on top of the Phase 32 model after it executes).

</domain>

<decisions>
## Implementation Decisions

### Cross-platform identity (LOCKED — user directive)
- Objects extracted from the Grasshopper canvas into the Computgraph MUST carry a DG ID that maps these objects in a cross-platform environment through the different connectors of Design Grammars.
- Counterpart objects across platforms share ONE DG ID within one Design State: a parametric wall object defined in Grasshopper and the BIM wall in Revit generated from that parametric wall MUST share the same DG ID.
- Mapped objects gain shared properties not available within a single platform. Worked example (LOCKED as the acceptance scenario): a facade panel representation in Revit is not aware of its insulation level; the Grasshopper panel representation calculates the insulation value with Ladybug components and writes it to the ComputGraph of Design Grammars — through the shared DG ID that property becomes available to the panel's Revit representation.

### Research mandate (LOCKED — user directive)
- The identification and mapping system MUST be elaborated from the state of the art, researched via Perplexity MCP. Two named study targets:
  1. **Rhino.Inside.Revit** — how it handles identification of objects between Grasshopper and Revit (Element Tracking: components remember/update the Revit elements they created; ElementId vs UniqueId; binding survival across document sessions).
  2. **Speckle** — how it handles identification of objects in a cross-platform environment (per-application `applicationId` vs content-hash `id`; how connectors match received objects to previously created native elements for update-in-place).
- Also survey the neighboring art the two targets sit in: IFC GlobalId (22-char base64 GUID), Revit UniqueId (episode GUID + ElementId), BHoM adapter id/fragment approach — the spec and ADR must position the DG ID design against these.

### Claude's Discretion
- Exact `dgId` format and minting algorithm (deterministic for convention-tagged Computgraph entities; relationship to Phase 32's `cg:<alg>:<kind>:<conventionName>` cgId; whether dgId wraps/extends cgId or is independent).
- Registry graph shape (dgId as property on Computgraph nodes + representation nodes vs a dedicated identity node label), as long as project isolation and the v4 `graph` property convention hold.
- Identity API surface and endpoint naming; error taxonomy follows the What+Where+How-to-fix house pattern.
- How DesignState/ObjState payloads reference dgId (statePayloadJson v2 extension vs sibling property) — must not break existing v1/v2 payload readers.
- Versioning: whether `dgId` lands in `cgContextJson v1` before Phase 32 executes or as a `v1.1` bump — Phase 32's versioned-contract rule applies either way.

</decisions>

<specifics>
## Specific Ideas

- The Frame worked example (Phase 32 fixture) is the natural test vehicle: its tagged entities should receive deterministic dgIds; the wall/panel scenario can be modeled with a `CgObject` + Emergent insulation `CgParameter`.
- "Within one Design State" matters: the GH↔Revit binding is scoped/observed through the DesignState that captured both representations — ObjState payloads are where dgId meets the v7.0 state pipeline.
- Provenance on every binding and shared-property write: platform, connector, timestamp (mirrors Phase 36's source/provider/model provenance discipline).
- Simulated Revit consumer for tests: bind a fabricated Revit UniqueId to a GH-extracted entity's dgId, then read the GH-written insulation property "as Revit" via the API — no Rhino running.

</specifics>
