**Core deterministic-hash pattern** (lines 8-19, 57-61, 108-112):
```csharp
/// <summary>
/// Deterministic ID generation for the DG state model hierarchy.
/// Every state entity ... gets a content-addressed StateId — identical inputs
/// produce identical StateId, enabling dedup across validation runs via MERGE
/// by StateId+project.
/// </summary>
public static class DesignStateIdGenerator
{
    private const string ObjectStatePrefix = "OS_";
    ...
    /// Produces an OS_-prefixed StateId: OS_&lt;SHA256(projectId + objectInstanceId + variableName)&gt;
    public static string ComputeObjectStateId(string projectId, string objectInstanceId, string variableName)
    {
        var input = $"{projectId}|{objectInstanceId}|{variableName}";
        return ObjectStatePrefix + HashToHex16(input);
    }

    private static string HashToHex16(string input)
    {
        var hash = SHA256.HashData(Encoding.UTF8.GetBytes(input));
        return Convert.ToHexString(hash)[..16];
    }
}
```

**Copy exactly:** the `HashToHex16` private helper (byte-identical), the `PREFIX + HashToHex16(pipe-joined-input)` shape, the doc-comment style stating the hash-input domain explicitly (this is what prevents cross-method hash collisions — note L94-95's explicit callout that `DS_` prefix is shared between two methods but input domains differ, so `DgIdMintingService` MUST use a domain-unique prefix `dg:` not reused elsewhere).

**Apply to `DgIdMintingService.Mint(project, definitionId, cgId)`:**
```csharp
private const string DgIdPrefix = "dg:";

public static DgId Mint(string project, string definitionId, string cgId)
{
    var input = $"{project}|{definitionId}|{cgId}";
    return new DgId(DgIdPrefix + HashToHex16(input));
}
```
Reuse the private `HashToHex16` verbatim (either duplicate the 5-line static method or, better, extract it to a small shared internal helper class if DG.Core conventions allow — `DesignStateIdGenerator` currently keeps its own private copy, so duplicating rather than refactoring matches the existing no-cross-file-coupling style).

    """Mint a new connector token: dgc_ + 32 bytes of url-safe randomness."""
    return TOKEN_PREFIX + secrets.token_urlsafe(32)

    """SHA-256 hex digest of a token. Only the hash is ever persisted."""

## Shared Patterns

### Deterministic content-addressed hashing
**Source:** `DG/src/DG.Core/Services/DesignStateIdGenerator.cs` lines 108-112 (`HashToHex16`)
**Apply to:** `DgIdMintingService.Mint()` — copy the SHA256 + `Convert.ToHexString(hash)[..16]` truncation verbatim; always pipe-join (`|`) ordered string inputs before hashing (see `ComputeObjectStateId` L59, `ComputePropStateId` L84-86).

### Structured error taxonomy (What+Where+How-to-fix)
**Source:** `data-service/app.py` line 583, `_structured_error_response(error, hint, code, status_code)`
**Apply to:** every new error path in `dg_identity.py`/`app.py`'s `/identity/*` routes — `error` states what went wrong and where (entity/dgId/native id), `hint` states how to fix it, `code` is a short SCREAMING_SNAKE identifier (`DGID_NOT_FOUND`, `DGID_AMBIGUOUS_BINDING`), `status_code` follows REST conventions (404 for not-found, 409 for conflict).

### Additive-only DTO evolution (no v1/v2 breakage)
**Source:** `DG/src/DG.Core/Serialization/DesignStatePayloadV2Serializer.cs` (existing camelCase `JsonSerializerOptions`, DTO shape) + the STATE.md-documented precedent of Phase 14 (Description coalesce) and Phase 17 (v1/v2 heuristic keyed on presence of `stateKind`/`objStates`)
**Apply to:** any DTO gaining `DgId` — new nullable property only, zero restructuring, zero changes to existing `Validate*` methods beyond the new field's own permissive check.

### Project-scoped Cypher (defense in depth)
**Source:** existing pattern throughout `data-service/app.py` (e.g. `read_single(query, {"graph": VALIDATION_GRAPH, "project": project, "runId": run_id})` at line 580) — every query threads `project` as a bound parameter, never string-interpolated.
**Apply to:** every Cypher statement in `dg_identity.py` — `MATCH`/`MERGE` patterns must include `project` in the match key (Pitfall 1 and Pitfall 4 in RESEARCH.md), on top of `project` already being folded into the `dgId` hash itself.

### Delegation shape: thin `app.py` route → domain helper module
**Source:** `app.py`'s `publish_validation()` delegating into `speckle_validation.py`'s `publish_validation_version()`/`build_validation_entity()`
**Apply to:** every `/identity/*` route in `app.py` — parse/validate the pydantic request, delegate to a `dg_identity.py` function, catch any domain exception and translate via `_structured_error_response()`.
