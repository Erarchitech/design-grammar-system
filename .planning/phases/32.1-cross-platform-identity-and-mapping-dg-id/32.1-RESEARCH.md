## Summary

This phase has no external-library risk — every deliverable is new logic layered on tools already vendored in this repo (System.Text.Json, SHA256, FastAPI, pydantic, the Neo4j driver). The research problem is entirely a **design** problem: pick a minting algorithm, a registry shape, and an API surface that are internally consistent with four precedents already living in this codebase — `DesignStateIdGenerator` (deterministic content-addressed hashing, PREFIX+HEX16), Phase 32's `cgId` (`cg:<alg>:<kind>:<conventionName>`, deterministic on convention name), the Phase 36 Computgraph publish/MERGE contract (dgId must survive re-publish), and the Connector Credential API (`connectors.py` mint/heartbeat/bind pattern with What+Where+How-to-fix errors) — while positioning the result against four external systems that have already solved variants of this problem: Rhino.Inside.Revit (element tracking, ElementId vs UniqueId), Speckle (content-hash `id` vs stable `applicationId`, proxy-based cross-cutting references), IFC (GlobalId, a 22-char compressed GUID baked into every `IfcRoot`), and BHoM (a separate `RevitIdentifiers` fragment carrying `PersistentId`/`ElementId`/category metadata, matched on push, not folded into the geometry object).

The clearest lesson across all four external systems is the same one this repo already learned the hard way in v2.0/v3.0 (the Var project-merge-key collision bug, `STATE.md` "carried from v3.0"): **identity and native-platform-id are different concerns, and conflating them causes either duplicate elements (Rhino.Inside without tracking) or silent cross-scope collisions (DG's own Var bug).** Every mature system studied keeps them separate: Speckle's `id` (content hash, changes) vs `applicationId` (stable, external); Revit's `ElementId` (session-local, unstable) vs `UniqueId` (durable); BHoM's geometry object vs its `RevitIdentifiers` fragment. DG ID should follow the same split: `dgId` is the durable, platform-neutral anchor (analogous to `applicationId`/`UniqueId`/BHoM's fragment `PersistentId`), and native ids (GH instance GUID, Revit UniqueId, IFC GlobalId, Speckle `applicationId`) are *representations bound to it* in a dedicated registry — never folded into or confused with the Computgraph entity's own content-addressed `cgId`.

**Primary recommendation:** Mint `dgId` as a SHA-256-derived, `dg:`-prefixed 16-hex id whose hash input is `project|definitionId|cgId` (extending — not replacing — Phase 32's `cgId`), following the exact `HashToHex16` pattern in `DesignStateIdGenerator.cs`; store it as an added property on every published Computgraph node; register platform bindings as a dedicated `Representation` node label (not properties on the entity node, since one entity can eventually bind N platforms) related via `HAS_REPRESENTATION`; and expose mint/resolve/bind/property-read-write through a new `data-service/dg_identity.py` that mirrors `connectors.py`'s persistence/response-model style and `app.py`'s `_structured_error_response()` error convention.

### Pattern 1: Deterministic content-addressed minting (extends `cgId`)
**What:** `dgId` is not independently minted — it is a SHA-256 hash over `project|definitionId|cgId`, hex-truncated to 16 chars and prefixed, exactly like every other id in `DesignStateIdGenerator`.
**When to use:** Every Computgraph entity that has a `cgId` (i.e., every `tagged`/`recognized` entity — never the untagged set).
**Example:**
```csharp
// Source: DG/src/DG.Core/Services/DesignStateIdGenerator.cs (existing precedent, pattern reused verbatim)
public static class DgIdMintingService
{
    private const string DgIdPrefix = "dg:";

    public static DgId Mint(string project, string definitionId, string cgId)
    {
        var input = $"{project}|{definitionId}|{cgId}";
        return new DgId(DgIdPrefix + HashToHex16(input));
    }

    private static string HashToHex16(string input)
    {
        var hash = System.Security.Cryptography.SHA256.HashData(
            System.Text.Encoding.UTF8.GetBytes(input));
        return Convert.ToHexString(hash)[..16];
    }
}
```
**Rationale for including `project` in the hash input (not relying on a composite `{dgId, project}` MERGE key alone):** `STATE.md`'s "Carried from v3.0" section records a real, previously-shipped bug — "Var merge key includes `project` — fixes latent v2.0 cross-project collision bug" — and `DesignStateIdGenerator.ComputeObjectStateId` already bakes `projectId` into its hash input specifically to avoid a repeat. Folding `project` into the `dgId` hash (rather than depending only on a project-scoped MERGE key, the pattern that caused the v2.0 bug for Var) is the more defensive of the two precedents already living in this codebase. `[VERIFIED: codebase — DesignStateIdGenerator.cs L57-61, STATE.md "Carried from v3.0"]`

**Rename/stability semantics (answers DGID-01's documented requirement):**
- Re-extracting the *same* annotated definition with *no* convention-name changes → identical `cgId` → identical `dgId`. This satisfies Success Criterion 1 directly.
- Renaming a convention group (e.g. `11_Var_SpansCount` → `11_Var_Count`) changes `cgId` (it is name-derived per Phase 32) → `dgId` re-mints by default. Document this as the default rule in `spec/DG-ID.md`.
- **Recommended stability escape hatch** (Rhino.Inside "Release" semantics as prior art, per the locked research mandate): if the minting service detects that the *member instance GUIDs* of a renamed entity exactly match a previously-minted `dgId`'s last-known member set, it may carry the old `dgId` forward and record the rename in an audit trail rather than silently re-minting — analogous to Rhino.Inside.Revit's `Enabled:Update` mode (match by prior binding, update in place) vs. `Release` (forget, mint new, causing duplicates on rerun). This is a **recommendation for planning discussion**, not a locked decision — `[ASSUMED]`, flagged in the Assumptions Log.

### Pattern 2: Representation registry as a dedicated node label, not flat properties
**What:** A `Representation` node (`graph:'Computgraph'`, `project`) with properties `platform` (`Grasshopper|Revit|IFC|Speckle`), `nativeIdKind` (`InstanceGuid|UniqueId|GlobalId|ApplicationId`), `nativeId`, `connector`, `boundAt`; related to its owning Computgraph entity via `HAS_REPRESENTATION`.
**When to use:** Every `bind` call (DGID-02).
**Example:**
```python
# Source: pattern mirrors connectors.py's CredentialCreatePayload/HeartbeatResponse pydantic shape
class BindRepresentationRequest(BaseModel):
    dg_id: str
    platform: str          # enum-validated: Grasshopper | Revit | IFC | Speckle
    native_id_kind: str    # enum-validated: InstanceGuid | UniqueId | GlobalId | ApplicationId
    native_id: str
    connector: str
    project: str

# Cypher (parameterized, project-scoped — DGID-05 requirement):
MERGE_BIND_CYPHER = """
MATCH (e {dgId: $dgId, project: $project})
MERGE (r:Representation {nativeId: $nativeId, platform: $platform, project: $project})
ON CREATE SET r.nativeIdKind = $nativeIdKind, r.connector = $connector, r.boundAt = datetime()
MERGE (e)-[:HAS_REPRESENTATION]->(r)
RETURN r
"""
```
**Why a dedicated label over flat properties:** Speckle's own architecture reaches the same conclusion from the opposite direction — "Proxies are stored at the Root Collection level and reference objects by their applicationId, not their id... allows relationships to persist even when object content changes" `[CITED: docs.speckle.systems/developers/data-schema/concepts]`. BHoM's `RevitIdentifiers` fragment is the same pattern under a different name — identity/correspondence metadata deliberately kept out of the geometry object `[CITED: bhom.xyz/.../BHoM%20vs%20Revit%20Identity/]`.

### Pattern 4: `SharedProperty` record for cross-platform property flow
**What:** A record (either a dedicated node label or a JSON-blob property on a lightweight anchor) keyed by `dgId` + `propertyName`, carrying `value`, `platform`, `connector`, `timestamp` — supports multiple writes across platforms without silent overwrite, per DGID-04's provenance requirement.
**When to use:** Any property computed on one platform that must be readable from another (the Ladybug-insulation worked example).
**Example:**
```python
# Source: pattern mirrors CGPD-03's provenance discipline (source/provider/model/timestamp per node)
WRITE_SHARED_PROPERTY_CYPHER = """
MATCH (e {dgId: $dgId, project: $project})
MERGE (sp:SharedProperty {dgId: $dgId, propertyName: $propertyName, project: $project})
SET sp.value = $value, sp.platform = $platform, sp.connector = $connector,
    sp.writtenAt = datetime()
MERGE (e)-[:HAS_SHARED_PROPERTY]->(sp)
"""
READ_SHARED_PROPERTY_CYPHER = """
MATCH (:Representation {project: $project})<-[:HAS_REPRESENTATION]-(e {dgId: $dgId})
      -[:HAS_SHARED_PROPERTY]->(sp:SharedProperty {propertyName: $propertyName})
RETURN sp.value AS value, sp.platform AS platform, sp.writtenAt AS writtenAt
"""
```
**Conflict policy direction (deferred full implementation per CONTEXT.md, but the spec must note direction):** `MERGE` + `SET` above is last-write-wins by design for the MVP. Document in `spec/DG-ID.md` that full bidirectional conflict resolution (e.g., per-platform authoritative-property mapping) is explicitly out of scope for 32.1 and deferred, consistent with the Deferred Ideas list.

### Pattern 5: Identity API surface mirrors the Connector Credential API's shape
**What:** `dg_identity.py` follows the exact module shape of `connectors.py` (module docstring explaining the security/persistence model, pydantic request/response models, small persistence helpers, thin `app.py` route wiring).
**When to use:** All of DGID-05.
**Example endpoint set:**
```
POST /identity/mint          {project, definitionId, cgId} -> {dgId}       (idempotent — safe to call from publish path)
GET  /identity/resolve       ?platform=&nativeId=&project= -> {dgId}       (native id -> dgId)
POST /identity/bind          {dgId, platform, nativeIdKind, nativeId, connector, project} -> Representation
GET  /identity/{dgId}/representations?project=             -> [Representation]
GET  /identity/{dgId}/properties?project=&propertyName=    -> [{value, platform, connector, writtenAt}]
POST /identity/{dgId}/properties  {propertyName, value, platform, connector, project} -> ack
```
**Error taxonomy (mirrors `_structured_error_response(error, hint, code, status_code)` in `app.py:583`):**
```python
# Source: data-service/app.py L583 (existing pattern, reused verbatim)
def _dgid_not_found_error(dg_id: str) -> HTTPException:
    return _structured_error_response(
        f"No Computgraph entity found with dgId '{dg_id}'.",
        "Verify the dgId was minted via /identity/mint or the publish path before resolving/binding.",
        "DGID_NOT_FOUND", 404,
    )

def _ambiguous_binding_error(native_id: str, existing_dg_id: str) -> HTTPException:
    return _structured_error_response(
        f"Native id '{native_id}' is already bound to dgId '{existing_dg_id}'.",
        "Detach the existing representation before re-binding, or confirm this is the intended object.",
        "DGID_AMBIGUOUS_BINDING", 409,
    )
```
DGID-05 explicitly forbids "silent misbinding" — the 409 above is the load-bearing check: `bind` must look up any existing `Representation` row for `(nativeId, platform, project)` before creating a new `HAS_REPRESENTATION` edge, and reject rather than silently repoint it to a different `dgId`.

### Anti-Patterns to Avoid
- **Minting `dgId` independently of `cgId`:** would require a second parallel deterministic-input contract to keep in sync with Phase 32's parser; every future change to the annotation grammar would need updating in two places. Wrap `cgId`, don't duplicate its logic.
- **Native ids as flat properties on the entity node:** blocks multi-platform binding (a wall bound to both Revit and IFC simultaneously) and loses per-binding provenance — see Pattern 2.
- **Storing the shared-property value only on the `Parameter` node with a plain `SET`:** destroys the write history needed for the conflict-policy note and for auditing "which platform actually computed this."
- **Treating `dgId` uniqueness as "global by construction" and skipping project-scoped MERGE keys anyway:** even with `project` folded into the hash, every Cypher `MATCH`/`MERGE` against a `dgId` must still include `project` in the pattern — defense in depth against exactly the class of bug `STATE.md` documents for Var.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Deterministic id generation | A bespoke GUID/counter scheme | SHA-256 hash of stable, ordered inputs (`DesignStateIdGenerator` pattern) | Guarantees re-extraction stability (DGID-01) for free — no persistence step required to "remember" the id between runs |
| Native-id ↔ dgId lookup | Ad hoc per-connector lookup tables (e.g. a JSON file per platform) | The Neo4j `Representation` registry, queried through one API | Every future connector (Revit, IFC, Speckle) must resolve against the *same* source of truth, or cross-platform identity silently forks |
| Cross-platform property provenance | Overwriting the property value in place with no audit trail | `SharedProperty` records with `platform`/`connector`/`writtenAt` per write | Matches the CGPD-03 provenance discipline already established for Computgraph nodes; needed for the conflict-policy note DGID-04 requires |

**Key insight:** Every external system surveyed for this phase (Rhino.Inside.Revit, Speckle, IFC, BHoM) independently arrived at "identity is a separate, durable token; native platform ids are representations bound to it, tracked in a registry/fragment/proxy, not folded into the object's own content" — this is not a DG-specific invention, it is the converged answer across the entire BIM/VPL interop space `[CITED: multiple, see Sources]`.

## Common Pitfalls

### Pitfall 1: Cross-project `dgId` collision (repeat of the v2.0 Var bug)
**What goes wrong:** Two different projects tag entities with the same convention name in structurally identical definitions (plausible — the Frame fixture itself is a worked example architects will likely reuse as a template) and end up computing colliding hash inputs if `project` is omitted.
**Why it happens:** `cgId` alone (`cg:<alg>:<kind>:<conventionName>`) carries no project scoping — it is deterministic *within* a definition, not globally.
**How to avoid:** Fold `project` into the `dgId` hash input (Pattern 1) *and* keep every registry query project-scoped in the `MATCH`/`MERGE` pattern — belt and suspenders, matching the actual fix applied to the Var bug in `migrations/2026-06-23_var_project_merge_key.cypher` (still pending live-run per `STATE.md` Blockers — a live reminder this class of bug is real and not yet fully closed even for the original case).
**Warning signs:** Two projects reusing the Frame fixture verbatim produce identical `dgId`s in a naive from-`cgId`-only scheme — test for this explicitly in the DgIdMintingService unit tests.

### Pitfall 2: Confusing Revit `ElementId` with `UniqueId` when designing the binding shape
**What goes wrong:** Storing the wrong Revit identifier as the durable `nativeId` for a `Representation` row — `ElementId` is an integer that "is not stable across upgrades and workset operations such as Save To Central, and might change" `[CITED: revitapidocs.com UniqueId Property page]`.
**Why it happens:** `ElementId` is the more commonly seen identifier in casual Revit API usage (it's what shows in most UI/debugging contexts); `UniqueId` requires deliberate lookup.
**How to avoid:** `nativeIdKind` for the Revit platform MUST default to `UniqueId` in the spec and any simulated-consumer test fixture, never `ElementId`. Document explicitly in `spec/DG-ID.md`'s survey table — this is exactly the distinction the locked research mandate calls out ("Revit UniqueId episode+ElementId").
**Warning signs:** A test fixture or `spec/DG-ID.md` example using a plain integer as a Revit `nativeId` value.

### Pitfall 3: Breaking v1/v2 `statePayloadJson` readers by restructuring instead of adding
**What goes wrong:** Adding `dgId` by changing the shape of `ObjStateDto`'s existing fields (e.g. nesting them under a new envelope) instead of adding a new optional field breaks the existing v1/v2 heuristic detection in the data-service and any consumer relying on the current flat shape.
**Why it happens:** It's tempting to "clean up" the shape while touching it.
**How to avoid:** Strictly additive: one new nullable property (`DgId`) on the DTO, no restructuring. Verify via the existing `DesignStatePayloadV2SerializerTests.cs` test suite (must remain green with zero changes required to existing test cases) plus one new test asserting a payload without `dgId` still deserializes.
**Warning signs:** Any diff to `DesignStatePayloadV2Serializer.cs` that touches `ValidateDesignState`/`ValidateDeserialized`/`ValidateParameter` beyond adding the new optional field's own (permissive) validation.

### Pitfall 4: Silent misbinding via idempotent `MERGE` on the wrong key
**What goes wrong:** A naive `MERGE (r:Representation {nativeId: $nativeId})` (without `platform`/`project` in the merge key) silently repoints an existing binding to a new `dgId` if the same raw native-id string happens to collide across platforms (e.g. a GUID reused between an IFC GlobalId's decoded form and a Revit UniqueId tail — low probability but non-zero, and exactly the "ambiguously bound" case DGID-05 calls out).
**Why it happens:** `MERGE` is convenient and idempotent-feeling, but its uniqueness key must be chosen deliberately.
**How to avoid:** The bind endpoint must explicitly `MATCH` first on `(nativeId, platform, project)`, compare the resolved `dgId` against the requested one, and return the 409 `DGID_AMBIGUOUS_BINDING` error (Pattern 5) rather than let a bare `MERGE` silently reassign.
**Warning signs:** Any Cypher in `dg_identity.py` that `MERGE`s a `Representation` without `platform` and `project` both in the merge-key property map.

## Code Examples

### Minting on the publish path (idempotent)
```python
# Source: pattern mirrors data-service/app.py's existing publish_validation() structure
@app.post("/identity/mint")
def post_identity_mint(payload: MintRequest):
    """Idempotent: re-minting the same (project, definitionId, cgId) always
    returns the same dgId — safe to call unconditionally from /computgraph/publish."""
    dg_id = DgIdMintingService.mint(payload.project, payload.definition_id, payload.cg_id)
    with driver.session() as session:
        session.run(
            "MERGE (e {cgId: $cgId, definitionId: $definitionId, project: $project}) "
            "SET e.dgId = $dgId",
            cgId=payload.cg_id, definitionId=payload.definition_id,
            project=payload.project, dgId=dg_id,
        )
    return {"dgId": dg_id}
```

|---------|--------|---------------------|
