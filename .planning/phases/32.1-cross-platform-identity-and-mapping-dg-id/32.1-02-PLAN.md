    1. **Purpose & scope** ‚Äî `dgId` is the durable, platform-neutral identity anchor; native platform ids are representations bound to it, never a stand-in for identity. State the GH-wall‚ÜîRevit-wall same-dgId-within-one-Design-State contract as normative (per CONTEXT locked decision).

    2. **Format & minting** ‚Äî `dgId` = the literal `dg:` prefix followed by the first 16 uppercase hex characters of SHA-256 over the UTF-8 bytes of `project|definitionId|cgId` (pipe-joined, that exact order). State that this hash-input contract is the SINGLE source of truth both `DG.Core.Models.Identity.DgIdMintingService` and the data-service `compute_dg_id` follow, and that a shared golden vector guards parity. State that `dgId` extends Phase 32's `cgId` (it hashes over it) rather than being independently minted. Also state the persistence seam as normative: the identity API's `mint_identity` anchors on the node match pattern `(cgId, definitionId, project)` ‚Äî Phase 36's publish MERGE key MUST include `cgId` (alongside `definitionId` + `project`, consistent with its CGPD-02 "definition id + convention name" contract) so pre-minted registry nodes and published Computgraph nodes coincide as ONE node, never duplicate.

    3. **Rename & stability rules** ‚Äî default: re-extracting an unchanged definition yields identical dgIds (determinism); renaming a convention group changes `cgId` and therefore re-mints the `dgId`. State the default rename-re-mints rule as the normative behavior for Phase 32.1. Note the member-instance-GUID carry-forward "stability escape hatch" (analogous to Rhino.Inside.Revit Update mode) as an explicit FUTURE refinement, out of scope for 32.1 ‚Äî cross-reference the ADR.

    4. **Collision policy** ‚Äî `project` is folded into the hash AND every registry Cypher query includes `project` in its match/merge key (belt-and-suspenders, per the shipped v2.0 Var collision lesson).

    5. **Binding model** ‚Äî `Representation` node (`graph:'Computgraph'`, `project`) with `platform` (`Grasshopper|Revit|IFC|Speckle`), `nativeIdKind` (`InstanceGuid|UniqueId|GlobalId|ApplicationId`), `nativeId`, `connector`, `boundAt`; related to the owning Computgraph entity via `HAS_REPRESENTATION`. State explicitly that the Revit representation uses `nativeIdKind = UniqueId`, never `ElementId` (ElementId is unstable across upgrades and workset operations). Attach/detach never rewrites `dgId`.

    6. **Shared-property semantics** ‚Äî `SharedProperty` node keyed by `dgId` + `propertyName` + `project`, carrying `value`, `platform`, `connector`, `writtenAt`; related via `HAS_SHARED_PROPERTY`. A property computed on one platform is readable from any bound representation keyed by `dgId`. State the conflict-policy DIRECTION: MVP is last-write-wins; full bidirectional per-platform authoritative resolution is deferred (cross-reference Deferred Ideas).

    7. **Partition decision** ‚Äî `Representation` and `SharedProperty` live under `graph:'Computgraph'` (not a new partition), minimizing the schema-propagation surface; note this is a locked Phase 32.1 decision resolving RESEARCH Open Question 2.

    8. **State-of-the-art summary table** ‚Äî a compact table positioning `dgId` against Rhino.Inside.Revit (ElementId/UniqueId + element tracking), Speckle (`id` content-hash vs stable `applicationId` + proxies), IFC (`GlobalId` 22-char compressed GUID in `IfcRoot`), Revit `UniqueId` (episode GUID + ElementId), BHoM (`RevitIdentifiers` fragment / `PersistentId`) ‚Äî one row each, with the "identity is separate from native id" convergent lesson stated. Cross-reference the ADR for full rationale and citations.
