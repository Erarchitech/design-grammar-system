    With the docker stack up and the V2 dev server running (`npm --prefix ui-v2 run dev` or `docker compose up -d design-grammars`), open the Reasoner screen and work through each verification step. Report pass/fail per step in `822-04-SUMMARY.md`.

    **Step A — Consistent:** Click **Run check** on the HermiT card. Confirm (a) a spinner + live elapsed-seconds counter appears while waiting, (b) the Run button is disabled during the run, (c) on completion the card shows the "✓ Schema consistent — no unsatisfiable classes found." verdict with a `Badge variant="solid"` "Consistent" badge, (d) the framing disclaimer ("Schema-consistency check…") is visible above the verdict, (e) a "Last checked <relative>" timestamp appears, and (f) the copy never says "valid/passed/compliant" (D-03). Record the actual elapsed time.

    **Step B — Inconsistent:** If the current fixture project has no unsatisfiable classes, this requires a temporary edit: add a known-inconsistent axiom to the test project's ontology (or use a pre-prepared inconsistent fixture). Rerun the check; confirm (a) the card shows "Schema inconsistent — N unsatisfiable classes." with the correct count, (b) each class is listed by its human `label` (never a raw IRI — D-02), (c) the `Badge variant="signal"` "Inconsistent" badge renders, and (d) the verdict copy says "schema inconsistent" never "schema invalid/failed" (D-03). If no inconsistent fixture is available, note this step as "SKIPPED — no inconsistent fixture" and record why in the SUMMARY.

    **Step C — Unknown (timeout):** Temporarily lower the sidecar timeout to force a semantic timeout: `docker compose run -e DG_REASONER_TIMEOUT_SECONDS=1 dg-reasoner` (or stub the data-service route to return a timeout body). Click Run check; confirm (a) the card shows the "Inconclusive" verdict with `Badge variant="outline"` (amber/achromatic, NOT green or red — criterion 3), (b) the copy is "– Inconclusive — the reasoner timed out before finishing. Try again.", (c) the UI never hangs/whitescreens during the wait, and (d) the "Last checked" timestamp is NOT set for timeout (only set for completed verdicts per Plan 03 Task 2).

    **Step D — Cancelled:** Start a run (against a valid project or a deliberately slow response). While the spinner is active, click **Cancel**. Confirm (a) the fetch aborts immediately, (b) the card shows "× Run cancelled — no result." in faint `--ink-a56` with no badge, (c) the Run button re-enables, and (d) the Cancelled state is visually quieter than Unknown (D-07/D-08 — a user-stop is never read as a reasoner verdict).

    **Step E — Pellet no-run:** Click the Pellet card (select it). Confirm (a) Pellet shows its "Integration pending" placeholder badge, (b) Pellet has NO Run button and NO "Run check" affordance (D-04, D-06), and (c) the "Selected — will be used when integration is complete" text (or its equivalent) still appears.

    **Step F — Re-run freshness (criterion 4):** After completing a run, trigger an ontology export change (e.g. add/remove a Class node in Neo4j via the Data screen or a Cypher query) and click Run check again. Confirm the new result reflects the change (not a cached/stale previous result — criterion 4). The "Last checked" timestamp updates.

    **Step G — Error line (D-09):** If feasible, stop the dg-reasoner container (`docker compose stop dg-reasoner`) and click Run check. Confirm a distinct error line ("Reasoner unavailable · …" + inline Retry, `--color-signal-ink`) renders — NOT folded into the Unknown verdict (D-09). Restart the container afterward.

    Record every step's pass/fail in `822-04-SUMMARY.md` under a "## Manual UAT Results" table with columns: Step | Behavior Verified | Result | Notes. If any step fails, file an issue in the SUMMARY with the observed behavior vs expected.
