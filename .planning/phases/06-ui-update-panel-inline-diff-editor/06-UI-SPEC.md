## Interaction Contract

### Update Knowledge Mode Selection

**Trigger:** User selects "Update Knowledge" from the knowledge mode dropdown in Specs&Notes tab.

**Behavior:**
- Sets `knowledgeMode = "update"`.
- Panel renders the Match view (prompt + submit button).
- No interference with Insert or Query Knowledge state — knowledge prompt text can be shared but update-specific state is isolated.

### Match View (`updateView === "match"`)

**Layout (top to bottom):**
1. Prompt textarea (`.prompt-input`, placeholder: "Describe what you want to change...")
2. Submit button: "Search Notes" — disabled when `knowledgeIsRunning` or prompt is empty
3. Progress/status area: `knowledgeStatus` text
4. Candidate list (appears after successful Match response):
   - "Select all / Deselect all" toggle link (12px, `.secondary-btn` styled as link)
   - Vertical list of `label.candidate-row` elements — each is a flex row:
     - `input[type="checkbox"]` — checked state tracked in `selectedNoteIds`
     - `span.candidate-title` — note title text (13px, weight 400)
     - `span.candidate-score` — relevance score formatted to 2 decimals (12px, `var(--muted)`)
5. "Edit Selected" button — disabled when `selectedNoteIds.length === 0`

**Empty state (D-04):** When Match returns 0 candidates, show inline message in the list area: "No matching notes found. Try a different description." — styled as `.hint`, no candidate rows rendered.

**Validation:** If `knowledgePromptText.trim() === ""`, set `knowledgeStatus = "Prompt is required"` and abort — no network call.

### Proposing View (`updateView === "proposing"`)

**Layout:**
1. "Back to search" button (`.secondary-btn`, top of view)
2. Candidate list — still visible but dimmed (`opacity: 0.4; pointer-events: none`) (D-14)
3. Step indicators (`.steps`) below dimmed list with update-specific labels (D-13):
   - "Fetching note content..."
   - "Generating edits..."
   - "Computing diff..."
   - "Complete"
4. Progress bar (`.progress`) filling at 25/50/75/100%
5. Status line showing current step text

**Note:** Propose is a **synchronous fetch** (not polling). Progress bar advances manually via `setKnowledgeProgress()`. No `startKnowledgePollingResult()`.

### Review View (`updateView === "review"`)

**Layout (top to bottom):**
1. "Back to search" button (`.secondary-btn`, `margin-bottom: 12px`)
2. Note header: **note title** (13px, weight 700) + counter "Note {N} of {total}" (12px, `var(--muted)`)
3. **If `hasChanges === true`:**
   - Collapsible diff panel (D-07):
     - Toggle link: "Show diff" / "Hide diff" (12px, `var(--accent)`, default: shown)
     - `.diff-preview` div with `dangerouslySetInnerHTML: { __html: diffHtml }`
     - Max height 240px, scrollable
   - Editable textarea (`.prompt-input`):
     - Pre-populated with `proposedContent` (D-08)
     - Value tracked in `editedContents[noteId]`
     - Auto-height: `min-height: 120px`, resize vertical
4. **If `hasChanges === false`:**
   - "No changes proposed" banner (`.hint`, italic) in place of diff panel
   - Textarea still shown with original content — user can manually edit or Skip
5. Navigation row (flex, `gap: 8px`, `margin-top: 12px`):
   - "Previous note" button — disabled when `currentNoteIndex === 0`
   - "Next note" button — disabled when `currentNoteIndex === updateDiffs.length - 1`; otherwise, transitions to summary when on last note (label changes to "Review Summary")
   - "Skip this note" button (`.secondary-btn`) — adds `noteId` to `skippedNoteIds`, advances to next note (D-10)

### Summary View (`updateView === "summary"`)

**Layout (top to bottom):**
1. "Back to search" button (`.secondary-btn`)
2. Heading: "Review Changes" (13px, weight 700)
3. Note list — each row is a flex container:
   - Note title (13px, weight 400)
   - Status badge:
     - "changed" — `color: var(--success)`, 12px, weight 600
     - "skipped" — `color: var(--muted)`, 12px, weight 400
4. "Confirm All" button — disabled when all notes are skipped (D-12)
   - Label: "Confirm All ({N} notes)" where N = non-skipped count
5. Status: "All notes skipped — nothing to confirm" if all notes skipped

### Done State

After successful Confirm:
- `knowledgeResponseText` set to `"Updated: [Title 1], [Title 2]"` in `var(--success)` color (D-15)
- Panel resets to Match view (`updateView = "match"`)
- All update state variables reset (candidates, selected, diffs, edited, skipped, noteIndex)
- `knowledgePromptText` cleared

### Error States

| Error | Display | Color |
|-------|---------|-------|
| Empty prompt | `knowledgeStatus = "Prompt is required"` | inherit (muted) |
| Match API error | `knowledgeStatus = "Error: {detail}"` | `var(--danger)` |
| Propose API error | `knowledgeStatus = "Error: {detail}"`, view resets to "match" | `var(--danger)` |
| Confirm 409 conflict | `knowledgeResponseText = "Conflict: Note was modified externally. Reload and retry."` | `var(--danger)` |
| Confirm other error | `knowledgeResponseText = "Error: {detail}"` | `var(--danger)` |
| Network failure | `knowledgeStatus = "Error: {message}"` | `var(--danger)` |

## Anti-Patterns Prohibited in This Phase

Sourced from RESEARCH.md. Executor must not:

1. **Use JSX syntax** — causes browser parse error; all elements via `React.createElement` alias `e`
2. **Use polling for Propose step** — `/knowledge/update/propose` is synchronous; use direct `await fetch()`, not `startKnowledgePollingResult()`
3. **Render all notes simultaneously in review** — D-09 mandates sequential one-at-a-time review
4. **Send `originalContent` or `proposedContent` on Confirm** — must send `editedContents[noteId]` (user's final text)
5. **Clear `knowledgePromptText` before Confirm** — prompt is needed for the Confirm payload
6. **Forget to reset all 8 update state variables** on "Back to search" or after Confirm — causes state leakage (RESEARCH.md Pitfall 2)
7. **Skip `.diff-del` / `.diff-ins` CSS declarations** — diff HTML renders but words appear unstyled (RESEARCH.md Pitfall 3)
8. **Enable "Edit Selected" with no checkboxes checked** — causes 400 from backend (RESEARCH.md Pitfall 5)
9. **Rebuild Docker without `--no-cache`** — stale `index.html` will be served

## Security Notes

- `dangerouslySetInnerHTML` is used for `diffHtml` — content originates from trusted backend (`data-service` on internal Docker network). Verify `word_diff_html` in `app.py` applies `html.escape()` on word tokens before wrapping in `<span>` tags (RESEARCH.md Open Question 1).
- No new auth, no new secrets, no new data ingestion paths. All calls go through existing Nginx proxy to existing endpoints.
