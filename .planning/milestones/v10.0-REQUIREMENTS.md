# Requirements: Design Grammar System v10.0 — Script Intelligence

**Defined:** 2026-07-08
**Status:** Future (isolated — pending v9.0 completion; do NOT activate while v9.0 is in flight)
**Core Value:** The connected LLM becomes an active partner on the Grasshopper canvas — generating parts of scripts, editing parts of scripts, and consulting the architect — grounded in the Computgraph, Design Rules mapped to script structure, and the v9.0 bridge/annotation foundations. Everything inherits the v9.0 safety contract: nothing touches the canvas or Neo4j without explicit on-canvas confirmation, all previews are undoable, tagged entities are ground truth.

> **Isolation contract:** This document and `v10.0-ROADMAP.md` are the only v10.0 artifacts. Nothing here modifies `.planning/phases/`, STATE.md, PROJECT.md's Current Milestone, or the paused v9.0 scope. Phase numbers are global (41–49); activation happens via `/gsd-new-milestone` after v9.0 ships.

## v10.0 Requirements

### Wire Substrate Persistence (WPER) — Phase 41

- [ ] **WPER-01**: The raw canvas substrate (`CgNode`/`CgWire`) persists per-node in Neo4j — `Node` label + wire relationships, `graph:'Computgraph'`, project-isolated, linked to the owning Pattern/Procedure — upgrading the v9.0 `cgContextJson` blob on `Algorithm`
- [ ] **WPER-02**: Substrate publishing is MERGE-idempotent on component instance GUID + definition id — re-publishing updates properties, never duplicates
- [ ] **WPER-03**: Topology is queryable via Cypher: upstream/downstream of a Parameter, wire path between two entities, fan-in/fan-out of a Pattern
- [ ] **WPER-04**: Two published snapshots of the same definition are diffable — an endpoint returns added/removed/changed nodes and wires (the substrate for edit previews and consulting)

### Component Knowledge Base (CKBS) — Phase 42

- [ ] **CKBS-01**: A versioned, DG-native knowledge base describes Grasshopper components — input/output parameters (names, types, access), defaults, and connection rules — covering at least the component set used by the reference definitions (Frame + one more real project)
- [ ] **CKBS-02**: The KB is queryable from data-service (search components, get component parameters — grasshopper-mcp tool equivalents) and injected into generation prompts via the context assembler
- [ ] **CKBS-03**: Connection validation uses the KB to check type/access compatibility before any wire command reaches the canvas

### Bridge Write Commands (BWRT) — Phase 43

- [ ] **BWRT-01**: The DG CANVAS LISTENER command table extends with write commands per the grasshopper-mcp reference: `add_component`, `connect_components`, `set_value` (slider/panel/toggle), `delete_component`, `wrap_group`
- [ ] **BWRT-02**: Every write batch executes inside a single `GH_UndoRecord` — one Ctrl+Z reverts the whole batch, partial application never survives failure
- [ ] **BWRT-03**: Write commands validate against the component KB before touching the canvas; invalid operations are rejected with structured errors (never a half-built block)
- [ ] **BWRT-04**: Writes are gated by an explicit "allow writes" toggle on the listener component, **off by default** — the v9.0 read+preview posture remains the default

### Cluster Introspection (CLIN) — Phase 44

- [ ] **CLIN-01**: The canvas serializer traverses GH Cluster internals (nodes, wires, nested annotation) into `cgContextJson` — clusters are no longer opaque
- [ ] **CLIN-02**: Clusters map to Procedures per the ontology ("Maps to a Grasshopper Cluster or Group"), their input/output hooks serializing as Interfaces
- [ ] **CLIN-03**: Recognition and structure validation operate on cluster internals the same as canvas groups; preview limitations inside clusters are documented, not silently skipped

### Script Part Generation (SGEN) — Phase 45

- [ ] **SGEN-01**: Given a rule or design-intent text plus a target slot (a tagged Procedure/Pattern or an empty canvas region), the LLM generates a node-block plan — components, values, wiring — validated against the component KB before anything reaches the canvas
- [ ] **SGEN-02**: The generated block arrives as an on-canvas preview (v9.0 proposal style: preview-styled group, ghost wiring into the live graph shown but not committed) — nothing wires into the live definition until confirmation
- [ ] **SGEN-03**: Confirming materializes the block with convention tags and provenance (rule/intent, provider/model, timestamp); rejecting removes every trace in one undo
- [ ] **SGEN-04**: Generation is grounded: only KB-known components are used; capabilities the KB lacks are reported as gaps — never invented component names

### Script Part Editing (SEDT) — Phase 46

- [ ] **SEDT-01**: Given an edit intent against an existing Computgraph entity, the LLM proposes a change set (add/remove/rewire/value change) rendered as an old→new diff — mirroring the v9.0 RING-02 atom-diff pattern at canvas level
- [ ] **SEDT-02**: The diff previews on the canvas (ghost wires/groups + a structured change report); confirming applies the whole change set in one undo record; cancelling leaves the definition untouched
- [ ] **SEDT-03**: After a confirmed edit, the Computgraph stays in sync — the flow prompts re-serialization/re-publish and the diff endpoint (WPER-04) shows exactly the applied changes

### Structure Rule Grammar (SRUL) — Phase 47

- [ ] **SRUL-01**: The structural constraint vocabulary grows beyond v9.0's `requiresProcedure`/`requiresParameter`: cardinality constraints, topology constraints (via WPER queries), and parameter-domain constraints
- [ ] **SRUL-02**: Structure rules become graph-native — stored as Metagraph extensions linked to Computgraph requirements — with the v9.0 JSON mapping file migrated losslessly
- [ ] **SRUL-03**: Structure-validation results persist as queryable run records (history), extending the v9.0 ephemeral reports

### Consulting Assistant (CONS) — Phase 48

- [ ] **CONS-01**: A multi-turn consulting session runs over the published Computgraph, Design Rules, and the live canvas (via the bridge) — session context persists across turns
- [ ] **CONS-02**: The assistant is rule-aware: "does this script satisfy R_URB_HEIGHT_MAX_75_V structurally?" answers by running the SRUL checks and citing their results and entities
- [ ] **CONS-03**: The assistant can *propose* actions — generate a block, edit an entity, run validation — handing off to the SGEN/SEDT/SRUL flows with their confirmation gates intact; it never executes canvas or graph mutations directly
- [ ] **CONS-04**: The consulting surface exists in ui-v2 (session panel pattern) and is reachable from Grasshopper (response readable GH-side via the bridge)

### Integration (INTG) — Phase 49

- [ ] **INTG-01**: E2E on the live stack: consult → generate a block into a tagged slot → confirm → re-publish → structure-validate → edit the block via diff preview → confirm → validate again — one session, no errors
- [ ] **INTG-02**: All v10.0 flows work on the Ollama fallback path (degraded quality acceptable; KB validation and confirmation gates fully enforced)
- [ ] **INTG-03**: CLAUDE.md, spec/, DG_OBSIDIAN, and the GH component reference document the write commands, KB, cluster support, structure-rule grammar, and consulting assistant; graphify refreshed

## Out of Scope

| Feature | Reason |
|---------|--------|
| Whole-definition synthesis from scratch | v10.0 generates *parts* into slots of an annotated definition; full-script synthesis needs a validated KB + generation track record first — future milestone |
| Autonomous canvas edits (no confirmation) | The confirm gate is the product's trust model — inherited from v9.0 RCGN-04/GHIN-04, non-negotiable |
| Bridge exposure beyond localhost | The canvas carries design IP; remote collaboration is a different milestone with a different threat model |
| Universal GH component coverage in the KB | KB grows from the reference-definition component set outward; plugin ecosystems (Kangaroo, LunchBox, …) added on demand |
| Multi-user consulting sessions | Single-architect sessions; the system remains single-team self-hosted |
| Fine-tuning / embeddings | Standing decisions carry forward: prompt + deterministic context assembly, no RAG unless proven insufficient |

## Key Decisions (v10.0 — proposed, confirmed at activation)

| Decision | Rationale |
|----------|-----------|
| Write commands extend the v9.0 listener's dispatch table | Phase 33 of v9.0 designed the table as the extension point; one listener, one protocol, one plugin |
| Writes off by default, explicit opt-in toggle | Read+preview stays the safe default posture; write capability is a deliberate per-session choice |
| Component KB is DG-native, seeded from reference definitions | grasshopper-mcp's KB proves the approach but its coverage/format aren't DG's; growing from real project component sets keeps quality measurable |
| Generation targets *slots* (tagged Procedure/Pattern), not blank canvases | Slots give the LLM bounded scope, the convention gives it naming, and the Computgraph gives it context — highest success probability |
| Editing = diff preview, same trust pattern as rules atom-diff (RING-02) and recognition preview (RCGN) | One mental model across the product: propose → preview → confirm |
| Structure rules go graph-native in this milestone | The v9.0 JSON file was the MVP escape hatch; consulting and editing need rules queryable next to the entities they constrain |
| Wire substrate persisted per-node (reverses v9.0 blob decision) | Editing diffs and topology-aware consulting need granular queries; v9.0 deferred exactly this pending a consumer — v10.0 is the consumer |
