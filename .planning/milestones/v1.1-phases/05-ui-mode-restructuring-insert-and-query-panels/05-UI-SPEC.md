## Interaction Contract

### Top-Level Tab Bar

**Placement:** Immediately after `gv-nav` back button and `h3` title, before the "Mode" label. Top margin: 14px (inherited from `.prompt-tabs` existing CSS — not a new Phase 5 value).

**Behavior:**
- Click "DesignRules" tab: sets `activeTab = "design-rules"`. Sidebar content area renders existing DesignRules modes (mode dropdown, Graph View select, prompt tabs, etc.)
- Click "Specs&Notes" tab: sets `activeTab = "specs-notes"`. Sidebar content area renders knowledge mode dropdown and selected knowledge panel.
- Tab state persists across all user actions. Switching tabs does not reset any prompt, response, or progress state in either tab (CONTEXT.md D-06).
- "DesignRules" is the default active tab (`activeTab` initializes to `"design-rules"`).

**Conditional visibility (CONTEXT.md D-07):**
- Graph View select (OntoGraph / Metagraph dropdown): render only when `activeTab === "design-rules"`
- "Clear the Graph" danger button: render only when `activeTab === "design-rules"`

### Knowledge Mode Dropdown

**Placement:** Within Specs&Notes tab content, same position as existing mode dropdown in DesignRules tab. Label text: "Mode".

**Options:**
- "Insert Knowledge" → `knowledgeMode = "insert"`
- "Query Knowledge" → `knowledgeMode = "query"`

**Behavior:** Replicates `.mode-dropdown` outside-click handler pattern (second `useEffect` on `knowledgeModeDropdownRef`, separate from existing `modeDropdownRef` handler — RESEARCH.md Pitfall 2).

### Insert Knowledge Panel (`knowledgeMode === "insert"`)

**Sub-tab bar placement:** Immediately below mode dropdown. Uses `.prompt-tabs` with two tabs: "From Folder" | "From Prompt". `insertTab` initializes to `"folder"`.

**From Folder sub-panel:**
- Text input (not textarea) with `placeholder="/repo/docs"` and `type="text"`
- `.hint` below input: "Accessible paths: any subfolder of the repository root. Example: DG_OBSIDIAN maps to /mnt/repo/DG_OBSIDIAN inside the service."
- Submit button label: "Import Notes"
- On submit: calls `POST /data-service/knowledge/ingest/folder` with `{project, path: folderPath.trim()}`
- Non-empty validation: if `folderPath.trim() === ""`, set `knowledgeStatus = "Path is required"` and abort — no network call
- On success: set `knowledgeResponseText = "Imported " + data.inserted + " notes" + (data.skipped > 0 ? " (" + data.skipped + " skipped)" : "")`; color: `var(--success)`
- On error: set `knowledgeResponseText = "Error: " + (data?.detail || res.status)`; color: `var(--danger)`
- This is a synchronous call — no polling, no step indicators, no progress bar for folder ingest

**From Prompt sub-panel:**
- Textarea with `.prompt-input` class; placeholder: "Describe what you want to record..."
- Submit button label: "Insert Knowledge"
- On submit: POST to `/n8n/webhook/dg/knowledge-ingest` with `{prompt: knowledgePromptText, project}`
- Triggers `startKnowledgePollingLatest("knowledge-ingest", "knowledge-ingest")` — knowledge-specific polling wrapper
- Step labels (RESEARCH.md Code Examples):
  1. "Send prompt"
  2. "Extracting title & tags"
  3. "Writing to graph"
  4. "Recording session"
  5. "Complete"
- On completion: `knowledgeResponseText = "Created note: " + (payload.title || payload.note_title || payload.noteId || "unknown")`; color: `var(--success)`
- Progress bar and step indicators rendered using `knowledgeProgress` and `knowledgeStep` state — separate from DesignRules `progress`/`step` (RESEARCH.md Pitfall 1)

### Query Knowledge Panel (`knowledgeMode === "query"`)

**Layout:** Mirrors existing Query Rules mode exactly (CONTEXT.md D-12). Elements in order:

1. Textarea with `.prompt-input` class; placeholder: "Ask a question about your project knowledge..."
2. Submit button label: "Query Knowledge"
3. Progress bar (`.progress` with `knowledgeProgress`)
4. Step indicators (`.steps` with `knowledgeStep`):
   1. "Send question"
   2. "Searching knowledge graph"
   3. "Generating answer"
   4. "Recording session"
   5. "Complete"
5. Response textarea with `.response-output` class (shows NL answer)
6. Workflow Cypher textarea with `.workflow-output` class (shows Cypher query used — REQUIREMENTS.md QRYK-02)
7. Status line (`.status`) with `knowledgeStatus`

**API:** POST to `/n8n/webhook/dg/knowledge-query` with `{prompt: knowledgePromptText, project}`; polls via `startKnowledgePollingLatest("knowledge-query", "knowledge-query")`.

### Async Polling (Knowledge Modes)

**State isolation:** All knowledge async state uses separate variables — `knowledgeIsRunning`, `knowledgeProgress`, `knowledgeStep`, `knowledgeStepTimes`, `knowledgeStatus`, `knowledgeResponseText`, `knowledgeResponseCypher`. These are never shared with DesignRules state variables (`isRunning`, `progress`, `step`, `responseText`, `responseCypher`) — RESEARCH.md Pitfall 3, Pitfall 4.

**Implementation pattern:** Thin wrapper function `startKnowledgePollingLatest(workflowKey, modeType)` calls the same underlying `fetch` loop infrastructure but writes to knowledge state setters. Do not modify the existing shared `startPollingLatest` / `startPollingResult` functions.

**n8n webhook auth:** Uses `cfg.n8nUser` / `cfg.n8nPassword` with Basic auth header — same pattern as `requestGrammar()`. Webhook paths are hardcoded (`/n8n/webhook/dg/knowledge-ingest`, `/n8n/webhook/dg/knowledge-query`) — not read from config.template.js (RESEARCH.md Pitfall 6).

## Anti-Patterns Prohibited in This Phase

Sourced from RESEARCH.md. Executor must not:

1. **Use JSX syntax** — causes browser parse error; all elements via `React.createElement` alias `e`
2. **Share `responseText`/`responseCypher`/`isRunning`/`progress`/`step` between tabs** — violates D-06; use `knowledge*` prefixed variables
3. **Modify existing `startPollingLatest` or `startPollingResult`** — write a new knowledge-specific wrapper instead
4. **Omit the outside-click handler for `knowledgeModeDropdownRef`** — dropdown will not close on outside click
5. **Use `&amp;` in React.createElement string arguments** — use literal `&` or `\u0026` for "Specs&Notes"
6. **Rebuild Docker without `--no-cache`** — stale `index.html` will be served
7. **Read `cfg.n8nWebhook` for knowledge webhooks** — use hardcoded paths `/n8n/webhook/dg/knowledge-ingest` and `/n8n/webhook/dg/knowledge-query`
8. **Show "Clear the Graph" in Specs&Notes tab** — confusing; hide when `activeTab === "specs-notes"`
