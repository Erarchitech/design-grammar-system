# Phase 823 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** This phase's ONLY UI surface is a new "SHACL Data Integrity" section inside `ModelScreen.jsx`'s right-hand Properties sidebar, in the `propMode === "run"` branch (the selected validation run's detail). No new screens, no new routes, no Reasoner-screen changes (822 scope). Grasshopper-side surfacing (`ErrorMessageTemplates.ShaclViolation`, VALIDATOR Report output) is a separate C# concern outside this UI contract's authority — mentioned here only where it constrains UI copy parity.

## Color

The project's color system (`colors.css`) is deliberately monochrome with exactly **one** chromatic hue reserved for failure/selection ("Color is absence, not expression"). SHACL's Solibri-style red/orange/yellow requirement (D-09/D-18) is an explicit, user-authorized extension of that system — **two new semantic tokens must be added** (warning-orange, info-yellow), following the exact structural pattern the existing `--color-signal` family already uses. This is the single most consequential design decision in this contract; flagged here for the checker and executor.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--surface-sidebar` (`#fafafa` / dark `#171717`) | Sidebar background hosting the new Panel (unchanged, inherited) |
| Secondary (30%) | `--surface-card` / `--color-paper` (`#ffffff` / dark `#1b1b1b`) | Panel and Collapsible surfaces (unchanged, inherited) |
| Accent (10%) | `--color-signal` (`#e7000b` / dark `#ff3b44`) — existing, unchanged | Reserved for: **violation-severity chip/text only** in this section (plus its pre-existing use on Failing-items/selection elsewhere on screen — no expansion of accent's existing meaning) |
| Destructive | `--color-signal-ink` (existing `--status-destructive` alias) | Not used in this phase — no destructive UI actions in the SHACL section (read-only display) |

### New tokens required (add to `ui-v2/src/styles/tokens/colors.css`)

Mirrors the existing `--color-signal` / `--color-signal-ink` / `--color-signal-soft` naming pattern exactly. Add immediately after the existing Signal Red block, both in `:root` and the `:root[data-theme="dark"]` override:

```css
/* SHACL severity — Solibri-style extension of the single-hue system.
   Two new chromatic hues, scoped exclusively to SHACL finding severity
   (never reused as a general-purpose accent elsewhere). D-09/D-18. */
--color-warning: #d97706;                       /* orange base — chip fill source, icons */
--color-warning-ink: #b45309;                    /* darker orange — text on light surfaces */
--color-warning-soft: rgba(217, 119, 6, 0.10);   /* chip background wash */

--color-info: #ca8a04;                           /* gold-yellow base (not pure #eab308 — insufficient text contrast) */
--color-info-ink: #854d0e;                        /* darker gold-brown — text on light surfaces */
--color-info-soft: rgba(202, 138, 4, 0.10);       /* chip background wash */
```

Dark-mode override block (append inside existing `:root[data-theme="dark"]`):

```css
--color-warning: #fb923c;
--color-warning-ink: #fdba74;
--color-warning-soft: rgba(251, 146, 60, 0.16);

--color-info: #facc15;
--color-info-ink: #fde047;
--color-info-soft: rgba(250, 204, 21, 0.16);
```

Severity → token mapping (D-09, exact):

| SHACL severity | Chip/text color | Base token | Ink token (text) | Soft token (chip bg) |
|---|---|---|---|---|
| `sh:Violation` → `violation` | red | `--color-signal` | `--color-signal-ink` | `--color-signal-soft` |
| `sh:Warning` → `warning` | orange | `--color-warning` | `--color-warning-ink` | `--color-warning-soft` |
| `sh:Info` → `info` | yellow | `--color-info` | `--color-info-ink` | `--color-info-soft` |

The "conforms" positive state and the "not checked" / "unavailable" quiet states use **no chromatic color at all** — `--text-muted` / `--color-ink` only, consistent with the project's "color is absence" philosophy: color is spent exclusively on findings that need attention, never on neutral/positive states.

## Component Contract

### `Badge` extension (`ui-v2/src/components/display/Badge.jsx`)

Add three new variants alongside the existing `solid`/`soft`/`outline`/`signal`, using the new tokens:

```js
violation: { background: "var(--color-signal-soft)", color: "var(--color-signal-ink)", border: "1px solid transparent" }, // == existing "signal" variant, semantic alias
warning:   { background: "var(--color-warning-soft)", color: "var(--color-warning-ink)", border: "1px solid transparent" },
info:      { background: "var(--color-info-soft)",    color: "var(--color-info-ink)",    border: "1px solid transparent" },
```

### `Collapsible` extension (`ui-v2/src/components/surfaces/Collapsible.jsx`)

The existing `signal` boolean prop only distinguishes red/neutral. Extend with a `tone` prop (`'violation' | 'warning' | 'info' | 'neutral'`, default `'neutral'`) that drives the count-badge background/color using the same three token pairs above. Keep `signal={true}` as a backward-compatible alias for `tone="violation"` — zero changes required at the two existing call sites (Failing/Passing items).

### New section layout (inside `ModelScreen.jsx`, `propMode === "run" && view` branch)

Insert a new `<Panel title="SHACL Data Integrity">` **directly after** the existing `<Panel title="Validation run">` block (after its `StatBlock` row) and **before** the `Failing items` `Collapsible`. Rationale: SHACL is a cross-cutting per-run result at the same priority tier as the run's own stat summary, and should be seen before drilling into SWRL rule/entity detail below it.

**State 1 — Conforms** (`findings.length === 0`, report present):
```
[✓ inline SVG check, 14px, stroke var(--color-ink)]  Conforms
"No data-integrity issues found in this run."   — 13px/400, var(--text-muted)
```
No color. This is the positive-but-quiet variant of the same visual language `ReasonerScreen`'s screen-level note uses (frosted note tone, muted body copy).

**State 2 — Not checked** (no `shaclReportJson` on the run — pre-823 runs):
```
"Not checked" — 13px/400, var(--text-muted), no icon, no border-left accent
"This run predates SHACL validation." — 11px/400, var(--text-muted)
```
Never rendered with `--color-signal` or any warning-adjacent styling — must read as neutral absence-of-data, not an error, matching D-17 verbatim and mirroring Phase 822's Unknown-state precedent for OWL consistency.

**State 3 — Unavailable / timeout** (`shaclReport.status` is `"unavailable"` or `"timeout"`):
```
"Not evaluated" — 13px/400, var(--text-muted)
"SHACL validator was unreachable when this run published." (status = unavailable)
"SHACL validation timed out when this run published." (status = timeout)
— 11px/400, var(--text-muted)
```
Same neutral, non-red treatment as State 2 — an infra hiccup is not a data-integrity finding and must not visually alarm the architect (D-02/D-17: SHACL is non-fatal and must never look like a failure of the run itself).

**State 4 — Findings present** (`findings.length > 0`):

Row 1 — severity chip summary (reproduces the "2 violations · 1 warning · 3 info" example literally, as colored `Badge` chips joined by a plain middot separator, only non-zero severities rendered, singular/plural noun agreement):

```
[Badge variant="violation"]2 Violations[/Badge]  ·  [Badge variant="warning"]1 Warning[/Badge]  ·  [Badge variant="info"]3 Info[/Badge]
```
Middot: `<span style={{ color: "var(--text-muted)" }}>·</span>`, 8px gap either side.

Row 2 onward — three `Collapsible` groups, one per severity, **omit a group entirely if its count is 0** (do not render an empty "0 Warnings" collapsible):

```
Collapsible label="Violations" tone="violation" count={n} open={true by default}
Collapsible label="Warnings"   tone="warning"   count={n} open={false by default}
Collapsible label="Info"       tone="info"      count={n} open={false by default}
```
Default open/closed states mirror the existing Failing-items(open)/Passing-items(closed) convention — the most severe, actionable group is expanded first.

**Finding card** (each item inside a severity `Collapsible`, replaces `CollapsibleItem` — needs a 3-line variant since findings carry more than primary/secondary):

```
┌─────────────────────────────────────────────┐
│ {focusLabel}                                  │  12px/500 mono, var(--color-ink)
│ {what}                                        │  13px/400 sans, var(--text-secondary)
│ Where: {where}                                │  11px/400 sans, var(--text-muted)
│ Fix: {howToFix}                               │  11px/400 sans, var(--text-muted)
└─────────────────────────────────────────────┘
```
Padding `8px 12px`, `borderBottom: 1px solid var(--color-hairline)` (last item: none), vertical gap `4px` between the four lines — matches `CollapsibleItem`'s existing padding/border exactly, extended to four lines instead of two.

**`shapeId` field:** present in the finding data object per D-10 but **MUST NOT be rendered anywhere in this UI**. It is an internal SHACL shape identifier (not raw `sh:*` vocabulary, but still implementation detail with zero meaning to an architect) — reserved for future dev-tooling/telemetry only. This is a hard constraint carried from D-10's "no raw RDF/SHACL vocabulary" bar.

### Data shape this UI reacts to (reference — not binding on backend field-naming discretion)

```
shaclReport: undefined                                          → State 2 (not checked)
shaclReport: { status: "unavailable" | "timeout" }               → State 3 (unavailable)
shaclReport: { findings: [] }                                    → State 1 (conforms)
shaclReport: { findings: [{severity, what, where, howToFix,
                            focusLabel, shapeId}, …] }            → State 4 (findings)
```

**Conforms vs. findings — do not conflate (hard rule):** the sidecar calls `pyshacl.validate` with `allow_infos=True` / `allow_warnings=True`, so pySHACL's `conforms` flag stays `true` when only info/warning findings exist — only `sh:Violation` findings flip it to `false`. The UI therefore keys states on **`findings.length`, never on a `conforms` boolean**:

- State 1 ("Conforms" positive copy) renders **only when `findings.length === 0`** — zero findings of any severity.
- State 4 renders whenever any findings exist, **including when `conforms` is still `true`** (only info/warning findings present). In that case the section shows the chip summary and groups as normal but must **not** display any "does not conform" / failure-toned heading — the chips themselves carry the severity story. No textual conforms/does-not-conform verdict label is rendered in State 4 at all; presence of warnings must never read as non-conformance.
- If the backend envelope includes a `conforms` field, the UI may ignore it entirely — it is redundant with the presence/absence of violation-severity findings and using it risks the exact conflation this rule forbids.

Exact field/envelope names are `gsd-planner`/backend discretion per CONTEXT.md's Claude's Discretion list ("shacl response envelope field names… whichever the Model screen consumes most naturally"). The four-state semantics above are the binding UI contract regardless of final field names — the executor should confirm the live shape against `modelApi.js`'s eventual extension and adjust field access, not the state logic or visual treatment.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | **None.** SHACL results are read-only and auto-populated per validation run (D-01/D-02 — server-side, synchronous, no UI-triggered re-run). The only interaction in this section is expand/collapse of severity groups (inherited `Collapsible` behavior, not a CTA). |
| Empty/positive state (conforms) | Heading: **"Conforms"** · Body: **"No data-integrity issues found in this run."** |
| Quiet state — not checked | **"Not checked"** · **"This run predates SHACL validation."** |
| Quiet state — unavailable | **"Not evaluated"** · **"SHACL validator was unreachable when this run published."** |
| Quiet state — timeout | **"Not evaluated"** · **"SHACL validation timed out when this run published."** |
| Findings summary (chip row) | **"{n} Violation(s)"** / **"{n} Warning(s)"** / **"{n} Info"** — singular noun at n=1, plural otherwise; "Info" itself does not pluralize |
| Finding — where prefix | **"Where: {where}"** |
| Finding — fix prefix | **"Fix: {howToFix}"** |
| Error state (transport failure loading the run itself — pre-existing, unrelated to SHACL) | Unchanged — this phase does not touch `ModelScreen`'s existing `loadErr` handling |
| Destructive confirmation | Not applicable — no destructive actions in this phase's UI scope |

**Tone constraint (D-10, hard rule):** every user-facing string in this section is architect-plain-language. `sh:focusNode`, `sh:sourceShape`, any `sh:*` term, and raw IRIs must never appear literally in copy or in `focusLabel`/`where`/`howToFix` values — those fields arrive pre-translated from `dg-reasoner` (D-10's label-enrichment pattern, reusing Phase 822's `unsatisfiable_classes` `{iri, label}` local-name-fallback approach). The UI's job is to render what it receives verbatim with no further scrubbing needed, but must not fall back to showing a raw IRI if a field is empty — fall back to the neutral literal string `"—"` (existing `KVRow` empty-value convention), never to a technical identifier.
