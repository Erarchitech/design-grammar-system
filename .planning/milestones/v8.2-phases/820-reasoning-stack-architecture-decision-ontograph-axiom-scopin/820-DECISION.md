# Phase 820: Reasoning-Stack Architecture Decision & OntoGraph Axiom Scoping -- Decision Record

## ADR-820-1: Hybrid Axiom-Scoping (Static TBox + Live Project Export + Curated Disjointness)

**Context:** OWL 2 DL reasoning (HermiT) needs real axioms (subClassOf, domain, range, disjointWith) to produce a non-trivial consistency answer. However, the live Neo4j OntoGraph is a flat vocabulary -- it declares terms (owl:Class, owl:DatatypeProperty, owl:ObjectProperty) but bears zero axiom-bearing relationship types (SUBCLASS_OF, DOMAIN, RANGE, DISJOINT_WITH) anywhere in the database. Exporting the live graph alone feeds the reasoner a declaration-only graph with no subsumption or disjointness, making any consistency check trivially pass -- a documented false positive.

Extending LLM ingestion (n8n webhook prompts) to emit axiom-bearing relationship types on rule ingest would fix the root cause, but carries a 9-file schema propagation cost (CLAUDE.md checklist) plus mid-milestone n8n prompt churn. This cost cannot be justified until the axiom requirements are proven in practice.

**Decision:** Union three sources at reasoning time:

1. **Static TBox** (`ontology/DesignGrammar-V7.owl`) -- the V7 meta-schema carries real axioms: 65 `rdfs:subClassOf`, 101 `rdfs:domain`, 110 `rdfs:range`, 75 `owl:Class` (rdflib-parsed counts, 2026-07-11). These provide the class hierarchy and property domain/range constraints.
2. **Live project-scoped OntoGraph + Metagraph export** -- the label-scoped Cypher extraction (scoped by node label, not the `graph` property, per `spec/LPG-OWL-MAPPING.md` Export Scoping) contributes per-project class/property/rule declarations.
3. **Curated `owl:disjointWith`** added once to the static TBox -- the V7 export has zero disjointness axioms, so consistency checking cannot meaningfully fail without them. Disjointness is a design-time artifact curated into the static TBox, never generated by the ingestion pipeline.

LLM-ingestion axiom emission is explicitly deferred out of v8.2.

**Consequences:**
- Reasoning becomes non-trivial (HermiT produces genuine consistency answers) without touching the ingestion pipeline.
- Disjointness is a curated design-time artifact, maintained in the static TBox alongside the meta-schema.
- The static TBox remains version-controlled and independently revisionable; live project data is per-export dynamic.
- The namespace separation (`dg/dgm/dgv/dgs/dgc:` meta-schema vs `ex:` per-project domain vocabulary, per `spec/LPG-OWL-MAPPING.md`) is preserved: the two namespace families are unioned only at reasoning time.

**Evidence:**

The spike executed against live `v8-ui-smoke` project data (Plan 01) produced the following evidence:

| Measure | Naive Export | Hybrid (Static TBox + Live + Curated) |
|---------|-------------|---------------------------------------|
| **Axiom source** | Flat live OntoGraph/Metagraph only (declaration nodes, no subClassOf/domain/range/disjointWith) | `DesignGrammar-V7.owl` TBox (65 subClassOf, 101 domain, 110 range, 0 disjointWith) + live project export + curated `ex:Door owl:disjointWith ex:Window` |
| **Seeded contradiction** | None | `ex:SlidingDoor rdfs:subClassOf ex:Door`, `ex:SlidingDoor rdfs:subClassOf ex:Window` (class under two disjoint parents) |
| **HermiT result** | `Inconsistent classes: []` (0 classes, trivially consistent) | `Inconsistent classes: [owl:Nothing, ex:SlidingDoor]` (2 classes -- `ex:SlidingDoor` detected as unsatisfiable) |
| **Verdict** | FALSE POSITIVE (the naive export cannot fail regardless of rule content) | NON-TRIVIAL (the hybrid approach catches meaningful TBox contradictions) |

The `owl:disjointWith` count in the static TBox is 0 (rdflib-parsed from `axiom_counts.txt`, 2026-07-11), confirming that even the curated TBox needs disjointness axioms added before any consistency check can fail -- the zero-disjointness baseline is the empirical justification for the curation step.

## ADR-820-2: dg-reasoner Sidecar Confirmed (Isolated Reasoning Service)

**Context:** The reasoning-stack architecture had a documented conflict between `STACK.md` (which described an embedded approach inside `data-service`) and `ARCHITECTURE.md` (which proposed a separate sidecar). This was recorded as a Pending Key Decision in `.planning/PROJECT.md` (line 134) awaiting Phase 820's determination. The sidecar approach was already research-recommended (see `.planning/research/` corpus) but needed empirical confirmation via the spike before flipping to a Shipped outcome.

**Decision:** Confirm the isolated `dg-reasoner` sidecar architecture (Owlready2/HermiT/Openllet + pySHACL running as a standalone service), not embedded in `data-service`. The sidecar is the target deployment model for Phase 822's reasoner integration.

**Consequences:**
- Isolates the JVM subprocess failure modes (hang, OOM, crash) from `data-service`'s Speckle-publish / validation-run hot path -- a crash in the reasoner subprocess cannot bring down the publish API.
- DL reasoning is worst-case exponential and the rule corpus grows unboundedly per project; a sidecar allows independent resource limits, scaling, and restart policies.
- The spike proved the isolation pattern is viable: HermiT ran as a local subprocess via Owlready2's bundled JAR (Java 17 required), and no blocking issue surfaced during the spike's execution.
- Phase 822 builds the production sidecar service; the spike's reproduction steps document the exact HermiT subprocess mechanics.

**Evidence:**

The spike (Plan 01) ran HermiT through Owlready2's `sync_reasoner()` API, which shells out to a bundled HermiT JAR via a local JRE subprocess. This is the exact isolation concern the sidecar addresses:

| Aspect | Spike Behavior | Production (Sidecar) Implication |
|--------|----------------|----------------------------------|
| HermiT invocation | `owlready2.sync_reasoner()` shells out to HermiT JAR (bundled inside Owlready2 package) | Sidecar wraps this same invocation, but in a separate process from `data-service` |
| JRE requirement | JRE 17 required on the runner | Sidecar Docker image includes `openjdk-17-jre-headless` |
| Memory exposure | HermiT runs as a subprocess of the Python script -- an OOM kills the script | Sidecar isolates: HermiT OOM kills only the sidecar, not `data-service` |
| No blocking issue | Spike ran both naive and hybrid exports through HermiT successfully (5 builtin-using rules stripped per HermiT's documented limitation) | Confirms the Owlready2 + HermiT integration path has no phase-blocking issues |

The spike's builtin-rule stripping (5 rules in `v8-ui-smoke`) confirms HermiT's documented limitation: builtin atoms are not supported. This is known and handled by filtering before reasoner input.

## Data-Quality Note: Systemic `graph`-Property Tagging Drift

During the live-data audit (2026-07-11), two instances of systemic graph-property mistagging were confirmed:

1. **`v8-ui-smoke` (spike project):** Two `Atom` nodes (`R_DOOR_ORIENTATION_V_A1`, `R_DOOR_ORIENTATION_V_A2`) carry `graph:'OntoGraph'` instead of the schema-mandated `graph:'Metagraph'`. Any export scoped by `{project, graph}` silently drops these atoms.
2. **`phase14-smoke` (broader dataset):** Six `DesignState` and two `Run` nodes carry `graph:'Metagraph'` instead of `graph:'ValidGraph'`.

**Status:** This is a documented known gap, not a repair task. The label-scoped export pattern mandated by `spec/LPG-OWL-MAPPING.md` (Export Scoping section) makes the drift harmless -- Cypher queries scope by node label, not the `graph` property. Phase 821's `ontology_export.py` inherits the same label-based discipline. No data-repair migration is scoped into Phase 820 (per Open Question 2 resolution).

**Assumption A3 note:** `v8-ui-smoke` is a QA/smoke-test project (as described in `.planning/STATE.md`). It was selected per CONTEXT.md's discretion rule ("pick the project with the most rules/atoms") -- it has 16 `Rule` nodes, the most of any project in the live database -- and `spec/LPG-OWL-MAPPING.md`'s live-data characterization confirmed its OntoGraph vocabulary (10 `Class`, 25 `DatatypeProperty`) is representative of real authored rules. This satisfies the success criterion's "real project's live OntoGraph data" wording: the spike proves the axiom-scoping approach against the project with the richest rule content, and the data-quality finding (mistagged atoms) is a real defect in live data, not a synthetic construction.
