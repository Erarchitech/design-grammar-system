<objective>
Build the testable, GH-independent core of the CONNECTOR platform-credential feature in DG.Core: a `ConnectorHeartbeatClient` that authenticates a platform token against data-service's `POST /connectors/heartbeat` and maps the HTTP result to a small `HeartbeatResult` (Authenticated / Rejected / Unreachable / NotAttempted), plus two new `ErrorMessageTemplates` entries for the in-canvas messages. All of this compiles without the Grasshopper SDK and is fully covered by `dotnet test`, so the phase's authentication and error-taxonomy behavior is provable independently of Rhino.

Purpose: Success Criteria 2 (a valid token authenticates a heartbeat) and 3 (an invalid/revoked/expired token yields a clear message, never a silent failure or crash) are grounded here as unit-tested Core logic. Wave 2 (Plan 02) wires this into `ConnectorComponent`.

Output: two new DG.Core source units + two new `ErrorMessageTemplates` methods + unit tests.
</objective>

| token value â†’ process memory/logs | The raw `dgc_` secret must stay in the Authorization header only; never persisted or logged |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-824-01 | Information Disclosure | token leaked into logs/returned objects | high | mitigate | HeartbeatResult is token-free (Task 1 verify); client uses the token only in the Authorization header, never logs it (Task 2 acceptance) |
| T-824-02 | Spoofing | forged/guessed token | low | accept | Authentication is delegated to data-service (SHA-256 hash match, non-revoked); the client only relays the token and honors 401 |
| T-824-03 | Denial of Service | data-service hang stalls the GH solve | medium | mitigate | 6-second timeout mirroring Neo4jConnectorService; timeout maps to Unreachable, never an unhandled hang/crash |
</threat_model>
