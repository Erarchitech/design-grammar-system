## Decisions Made

- **Sync `httpx.post()` over an async client:** every existing route in `data-service/app.py` is a sync `def` handler (including the sibling `/reasoner/settings` routes this lands beside); introducing `async def` + `httpx.AsyncClient` for one route alone would be an inconsistent, unnecessary architectural shift. `httpx.post(..., timeout=httpx.Timeout(...))` matches the file's prevailing style and `llm_gateway.py`'s existing sync `httpx.Client(timeout=...)` pattern.
- **Short-timeout semantics:** the proxy's `read=5.0` bound is deliberately much shorter than the sidecar's own `DG_REASONER_TIMEOUT_SECONDS=90` HermiT wall-clock bound. This is intentional per D-06/D-12 — the proxy's job is to fail fast on a genuinely unreachable/hung sidecar, not to babysit a real (potentially minutes-long) reasoning run; in the live test the real round-trip actually completed in 1.4-2.2s, well inside the bound, so no premature-timeout risk was observed against real data.
- **Direct-import integration tests over HTTP-based tests:** the new test file calls `reasoning.run_consistency(...)` and `ontology_export.build_graph(...)` directly (as `test_routes.py` does for the always-on tier), rather than making HTTP calls to a running sidecar container from inside the test process. This needed no new test-only HTTP client dependency and runs identically whether invoked via `pytest` inside the container or (with Neo4j/JRE reachable) from a CI runner with the same image.
- **Rebuilt both Docker images before verifying:** discovered mid-plan that `dg-reasoner`'s running container still had Plan 01's stub `app.py` (no `reasoning.py` at all) — the compose services hadn't been rebuilt since Plans 02/03 landed. Rebuilding both `dg-reasoner` and `data-service` (`docker compose build` + `up -d`) was necessary before either task's live verification could mean anything; this is captured as a deviation below since it wasn't an explicit plan step.
