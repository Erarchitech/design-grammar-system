## Overview

v10.0 turns the v9.0 canvas/Computgraph foundations into an active design partner. The build order runs **substrate → capability → intelligence**:

1. **Substrate** (Phases 41–44): granular wire persistence with topology queries and snapshot diffing; a DG-native component knowledge base; bridge write commands (off by default, KB-validated, undo-batched); cluster introspection.
2. **Capability** (Phases 45–47): script part generation into tagged slots with on-canvas preview/confirm; script part editing as old→new canvas diffs; the structure-rule grammar going graph-native with persisted validation history.
3. **Intelligence** (Phases 48–49): the multi-turn, rule-aware consulting assistant that proposes (never executes) actions; E2E + docs.

Every flow inherits the v9.0 trust model: propose → preview on canvas → confirm → apply/publish, all undoable, tagged entities as ground truth.

**9 phases (41–49), estimated ~12–16 implementation sessions.**

## Phases

### Phase 41: Wire Substrate Persistence and Topology Queries

**Goal:** The raw node/wire substrate of a serialized definition lives in Neo4j per-node — topology is queryable and two snapshots of a definition are diffable.

**Requires:** v9.0 Phase 36 (Computgraph persistence — this extends its publish path).

**Requirements:** WPER-01, WPER-02, WPER-03, WPER-04

**Deliverables:**
- `data-service` publish extension: `Node` label (instance GUID, component GUID, name, nickname, position) + `WIRE_TO` relationships (param-level endpoints), linked `MEMBER_OF` → owning Pattern/Procedure; `graph:'Computgraph'`, project-isolated
- MERGE keys: instance GUID + `definitionId` + `project`; snapshot versioning property (`snapshotAt`) supporting diff queries
- Topology query endpoints: upstream/downstream of a Parameter, path between entities, fan-in/fan-out of a Pattern
- `POST /computgraph/diff` — added/removed/changed nodes and wires between two snapshots of one definition
- Volume guardrail: substrate persistence per-definition opt-in flag if node counts threaten graph size (measure on the reference projects first)

**Success Criteria:**
1. Publishing Frame persists its nodes and wires; `MATCH` traversals answer "what feeds `11_Var_HTotal` downstream" correctly
2. Re-publishing unchanged Frame changes zero counts; publishing after moving one wire yields a diff naming exactly that wire
3. ui-v2 and existing v9.0 queries are unaffected (labels additive, no schema breakage)

### Phase 42: Component Knowledge Base

**Goal:** data-service knows what Grasshopper components exist, their parameters, and what can legally connect to what — the reliability substrate for every write command and generation prompt.

**Requires:** Nothing in v10.0 (parallel-safe with Phase 41).

**Requirements:** CKBS-01, CKBS-02, CKBS-03

**Deliverables:**
- `llm/gh_component_kb.json` (versioned) — per component: GUID, name, category, inputs/outputs (name, type, access item/list/tree), defaults, connection notes; seeded by **extracting from the live canvas via the bridge** (a `dump_component_info` listener command reading `IGH_Component` metadata beats hand-curation) for the component set of the reference definitions
- `data-service/gh_component_kb.py` — lookup API: search by name/keyword, get parameters, check connection compatibility (type/access rules)
- Context-assembler integration: KB excerpts injected into generation prompts (`dg_context.py`)
- KB gap report: components present on a canvas but absent from the KB are listed, not guessed

**Success Criteria:**
1. Every component in the Frame definition resolves in the KB with correct input/output parameter lists
2. `check_connection(Curve→Number)` style incompatibilities are rejected with the reason; compatible pairs pass
3. The KB grows without code changes (data file + extraction command), and its version is stamped into generation provenance

### Phase 43: Bridge Write Commands

**Goal:** The DG CANVAS LISTENER can build and modify canvas content — safely: KB-validated, undo-batched, and only when the architect has explicitly enabled writes.

**Requires:** Phase 42 (KB validation); v9.0 Phase 33 (listener dispatch table).

**Requirements:** BWRT-01, BWRT-02, BWRT-03, BWRT-04

**Deliverables:**
- Listener command-table extension (grasshopper-mcp reference semantics): `add_component` (by KB id/name, position), `connect_components` (param-level, KB-validated), `set_value` (slider/panel/toggle), `delete_component`, `wrap_group` (convention-named via the v9.0 name factory)
- Batch execution: one `{type: "apply_batch", parameters: {ops: [...]}}` command → single `GH_UndoRecord`; any op failure rolls back the whole batch
- `AllowWrites` boolean input on DG CANVAS LISTENER, **default false**; write commands received while false return a structured refusal
- data-service `gh_bridge.py` write wrappers + `gh_apply_batch` MCP tool; every batch logged (op list, origin, timestamp)

**Success Criteria:**
1. A scripted batch (add slider + add Line SDL + connect + wrap group) lands on the canvas correctly and one Ctrl+Z removes it all
2. A batch containing one invalid connection applies nothing (atomic rollback verified)
3. With AllowWrites off, every write command is refused with the What+Where+How-to-fix message; reads still work

### Phase 44: Cluster Introspection

**Goal:** GH Clusters stop being opaque — their internals serialize, recognize, and validate like canvas groups, and clusters map to Procedures per the ontology.

**Requires:** v9.0 Phase 32 (serializer). Parallel-safe with Phases 42–43.

**Requirements:** CLIN-01, CLIN-02, CLIN-03

**Deliverables:**
- `CanvasContextExtractor` extension: traverse `GH_Cluster` documents (nested `GH_Document`), serialize internal nodes/wires/annotations into `cgContextJson` (schema bump → `cg-context-2`, additive)
- Mapping: cluster → `Procedure` (ontology comment: "Maps to a Grasshopper Cluster or Group"); cluster input/output hooks → `Interface` entities
- Recognition + structure validation over cluster internals; documented preview limitation (GH cannot render preview groups *inside* a collapsed cluster — proposals for cluster internals render in the report and, when the cluster is opened for editing, on its internal canvas)

**Success Criteria:**
1. A definition with a clustered procedure serializes with the cluster's internal patterns/parameters visible in `cgContextJson`
2. The cluster appears in the published Computgraph as a Procedure with Interfaces matching its hooks
3. Structure validation flags issues inside clusters (e.g. untyped parameter) with the cluster path in the entity reference

### Phase 45: Script Part Generation

**Goal:** "Generate a footer configuration for this frame" produces a real, KB-validated node block — previewed on the canvas in the v9.0 proposal style and materialized only on confirmation.

**Requires:** Phases 42–43 (KB + write commands); v9.0 Phase 35 (preview/confirm pattern).

**Requirements:** SGEN-01, SGEN-02, SGEN-03, SGEN-04

**Deliverables:**
- `data-service/cg_generation.py` + `POST /computgraph/generate` — rule/intent + target slot (tagged Procedure/Pattern id or empty region) + Computgraph context + KB excerpts → **block plan JSON** (components, values, wires, group structure), schema-validated with bounded retry; every referenced component must resolve in the KB (SGEN-04 — gaps reported, never invented)
- Preview path: block plan compiles to a preview batch — components placed in preview-styled group, wiring into the live graph rendered as the proposal report + ghost markers, no live-graph connections until confirm
- Confirm path: DG STRUCTURE CONFIRM extension — accept compiles the block plan to an `apply_batch` (Phase 43), tags the result with convention names, records provenance (intent/rule, provider/model, KB version, timestamp); reject removes the preview in one undo
- Regression fixture: "generate 12_Proc footer" against a Frame stripped of its footer procedure — compared against the reference structure

**Success Criteria:**
1. The footer-regeneration fixture produces a block whose Computgraph shape (patterns/parameters/interfaces) matches the reference procedure's shape
2. Nothing connects into the live definition before confirmation; reject leaves the canvas byte-identical (undo-verified)
3. A request needing a component absent from the KB returns the gap report instead of a fabricated component

### Phase 46: Script Part Editing

**Goal:** "Change the truss to 8 spans and split the top chord differently" becomes a reviewable old→new canvas diff — applied atomically on confirm, never silently.

**Requires:** Phases 41 (diff substrate), 43 (write commands), 45 (block-plan compiler reused for the "new" side).

**Requirements:** SEDT-01, SEDT-02, SEDT-03

**Deliverables:**
- `data-service/cg_editing.py` + `POST /computgraph/edit` — edit intent + target entity + Computgraph/topology context → **change-set JSON** (`add`/`remove`/`rewire`/`set_value` ops with before/after), schema-validated, KB-checked
- Canvas diff preview: removed elements ghost-styled, added elements preview-styled, rewires as annotated markers + structured change report (GH panel + ui-v2)
- Confirm → single `apply_batch`; cancel → preview cleared, definition untouched
- Sync loop: after a confirmed edit, prompt re-serialization/re-publish; `POST /computgraph/diff` (Phase 41) must show exactly the applied change set (closed-loop verification)

**Success Criteria:**
1. An edit intent produces a change set whose canvas preview matches the report; confirm applies it in one undo record
2. Cancel after preview leaves the definition and Computgraph unchanged
3. Post-edit re-publish diff equals the confirmed change set — no hidden drift

### Phase 47: Structure Rule Grammar (graph-native)

**Goal:** Structural constraints become first-class graph citizens — richer vocabulary, stored next to the rules they extend, with persisted validation history.

**Requires:** Phase 41 (topology constraints query WPER); v9.0 Phase 37 (MVP checks + JSON mapping).

**Requirements:** SRUL-01, SRUL-02, SRUL-03

**Deliverables:**
- Constraint vocabulary v2: cardinality (`exactly/atLeast/atMost` procedures/patterns/parameters), topology (must-feed, must-not-cycle, interface-connectivity — via Phase 41 queries), parameter-domain (dataType, domain bounds vs rule limits)
- Graph-native storage: structure requirements as Metagraph extension nodes linked `CONSTRAINS` → Computgraph entity selectors; lossless migration of the v9.0 `llm/structure_rules.json` + the migration script
- Validation history: structure-validation runs persist as lightweight run records (reusing the `Run` conventions), queryable per definition/project
- Schema propagation checklist for the new labels/relations

**Success Criteria:**
1. Every v9.0 JSON-mapped check passes identically after migration (regression-verified)
2. A topology constraint ("HTotal must feed the top-chord pattern") evaluates via graph queries and flags a rewired violation
3. Validation history answers "when did this definition last pass rule R_X's structural checks?"

### Phase 48: Consulting Assistant

**Goal:** A multi-turn, rule-aware conversation about the script — grounded in the Computgraph, live canvas, and Design Rules — that can propose generation/editing/validation actions but never executes them itself.

**Requires:** Phases 45–47 (the actions it proposes); v9.0 Phase 37 `/consult` (single-shot seed).

**Requirements:** CONS-01, CONS-02, CONS-03, CONS-04

**Deliverables:**
- `data-service/cg_consulting.py` — session store (multi-turn context: prior turns + Computgraph scope + live-canvas pulls via bridge), `POST /computgraph/consult/session`
- Rule-aware answers: structural questions trigger SRUL checks and cite results/entities; topology questions use Phase 41 queries — grounded-citation post-check carried from v9.0
- Action proposals: responses may include typed action suggestions (`generate`, `edit`, `validate`) with prefilled payloads that hand off to the Phase 45/46/47 flows — confirmation gates intact, no direct execution (CONS-03)
- Surfaces: ui-v2 session panel (Graph screen pattern) + GH-side readable responses via the bridge
- Works on Ollama fallback

**Success Criteria:**
1. A three-turn session (structure question → rule check → "then generate the fix") holds context and ends with a prefilled generation proposal — not an executed change
2. "Does this script satisfy R_URB_HEIGHT_MAX_75_V structurally?" runs the mapped checks and cites their entities
3. Killing the LLM provider mid-session degrades to Ollama without losing session state

### Phase 49: E2E Validation and Docs

**Goal:** The full script-intelligence loop runs end-to-end on the live stack, and every new capability is documented.

**Requires:** Phases 41–48.

**Requirements:** INTG-01, INTG-02, INTG-03

**Deliverables:**
- E2E drill: consult → generate block into tagged slot → confirm → re-publish → structure-validate → edit via diff preview → confirm → re-validate — one session, live Docker + Rhino
- Ollama-fallback regression across all v10.0 flows (KB validation and confirm gates enforced)
- Docs: CLAUDE.md (service map, schema tables, gotchas), `spec/DATABASE.md` (substrate + structure-rule labels), DG_OBSIDIAN decision notes, GH component reference (listener write mode, confirm flows), graphify refresh

**Success Criteria:**
1. The E2E drill completes without errors; every canvas mutation in the drill was preceded by an on-canvas confirmation
2. All 31 v10.0 requirements checked off with traceability complete
3. A new user can enable writes, generate a block, and revert it using only the docs
