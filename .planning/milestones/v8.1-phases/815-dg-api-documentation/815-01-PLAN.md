# Plan 815-01: DG API Documentation

## Objective

The DG API Docs region hosts a Revit-API-style documentation browser for developers building new connectors: a navigable tree (sections → endpoints/classes → members) with a detail pane, driven by structured content files so new pages need zero viewer-code changes.

## Context (read before coding)

- Structural reference: Revit API documentation — left navigation tree, per-member pages with Syntax / Parameters / Return Value / Remarks / Examples blocks. Reproduce that information architecture, styled with the V2 design system.
- Document the REAL API — read `data-service/app.py` (connector endpoints from Phase 812, validation publish endpoints, LLM gateway endpoints) and `data-service/connectors.py` (registry, token format `dgc_`, heartbeat auth) so request/response examples are accurate, not invented.
- `ui-v2/src/screens/ApiDocsScreen.jsx` — skeleton from Phase 810; fill it in.
- Design system as in sibling plans; monospace via `--font-mono` (Geist Mono) for code blocks; no new shared primitives in `components/`.

## Tasks

### Task 1: content architecture (APID-03)

`ui-v2/src/screens/apidocs/content/` — one module per section (e.g. `01-getting-started.js`, `02-authentication.js`, `03-credentials-api.js`, `04-heartbeat-api.js`, `05-connectors-status-api.js`, `06-validation-publish-api.js`, `07-extending.js`) each exporting a structured object `{id, title, pages: [{id, title, summary, blocks: [...]}]}` with typed blocks (`text`, `code`, `endpoint` (method+path+params+request+response), `table`, `note`). An `index.js` aggregates via `import.meta.glob('./##-*.js', {eager: true})` sorted by filename — adding a file = new section, no viewer change. Include a "Extending this documentation" page that documents the content-module format itself.

### Task 2: viewer (APID-01)

Tree pane (sections → pages, expandable, active highlight per V2 selection grammar) + detail pane rendering the block types (endpoint blocks Revit-API-style: signature line, parameter table, request/response examples in mono code blocks). Breadcrumb header. Keep state local to the screen.

### Task 3: content (APID-02)

Author accurate content for: base URL/proxy paths (`/data-service/...` via nginx), credential lifecycle (create → copy-once token → paste into connector component), heartbeat contract (`Authorization: Bearer dgc_...`, status semantics never_connected/active/stale), status querying, validation publish flow (existing endpoints incl. request/response JSON), and error-response format. Examples as curl + JSON.

### Task 4: Verify

`npm --prefix ui-v2 run build` passes; tree navigation works in dev; every endpoint documented exists in app.py (cross-check paths).

## Verification criteria

Success criteria 1–3 of Phase 815 in `.planning/ROADMAP.md`.
