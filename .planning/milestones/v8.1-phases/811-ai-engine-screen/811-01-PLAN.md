# Plan 811-01: AI Engine Screen

## Objective

The AI Engine region is a working LLM setup screen: provider selection (Anthropic / OpenAI / Ollama), model selection, manual API-key input (stored encrypted via the existing gateway, never echoed back), current-config status, and a connection test with clear success/failure feedback.

## Context (read before coding)

- Backend already exists — do not rebuild it: `GET/PUT/DELETE /llm/settings` and `POST /llm/settings/test` in `data-service/app.py` (~line 912), implemented over `data-service/llm_gateway.py` (Fernet-encrypted keys; response exposes `apiKeyConfigured`, never the key). Read both to learn the exact request/response shapes, including how provider/model/baseUrl are represented and whether model discovery (`list_models`) is exposed via an endpoint.
- `ui-v2/src/screens/AiEngineScreen.jsx` — skeleton created by Phase 810; fill it in.
- `ui-v2/src/lib/` — follow the existing API-module conventions (e.g. `modelApi.js`, `graphApi.js`) for fetch style and base paths. nginx proxies `/llm/` in production (`ui-v2/nginx.conf` line 23); check `ui-v2/vite.config.js` dev proxy and add `/llm` if missing.
- Design system: `ui-v2/src/components/` primitives (Button, Input, Select if present), `dg-frost` cards, `dg-annotation` eyebrows. Do NOT add new shared primitives to `components/` — keep any new pieces local to the screen file.

## Tasks

### Task 1: `ui-v2/src/lib/llmApi.js`

`getSettings()`, `saveSettings(payload)`, `testConnection()`, plus model listing if the backend exposes it. If the gateway has `list_models` adapters but no HTTP endpoint, add a minimal `GET /llm/models` endpoint to `data-service/app.py` (provider from persisted settings; structured JSON errors per app.py conventions) — otherwise fall back to a static curated model list per provider in the UI and note it in the SUMMARY.

### Task 2: AiEngineScreen UI

- Provider selector (Anthropic / OpenAI / Ollama) and model selector for the chosen provider (AIENG-01)
- API key input (`type="password"`), submitted only when the user typed a new value; after save, field clears and a "key configured" badge reflects `apiKeyConfigured` (AIENG-02, AIENG-03)
- Active-config summary block: current provider, model, key status (AIENG-03)
- "Test connection" button → `POST /llm/settings/test`; render success or the gateway's error message in actionable form (AIENG-04)
- Loading/saving/error states; V2 look (frost card sections, annotation eyebrows, pill geometry)

### Task 3: Verify

`npm --prefix ui-v2 run build` passes. If the data-service is reachable (docker compose service on :8080 or direct :8000), exercise get→save→test round-trip; otherwise verify against the response shapes from code reading and say so in the SUMMARY. If you touched app.py, run `python -m pytest data-service/tests/ -q`.
