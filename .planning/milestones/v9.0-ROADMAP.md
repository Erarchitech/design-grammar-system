# Roadmap: Design Grammar System v9.0 — AI Workflow Intelligence

**Planned:** 2026-07-03
**Status:** Future (isolated — pending v7.0 completion; do NOT activate while v7.0 phases 14–20 are in flight)
**Depends on:** v7.0 (VALIDATOR rework, composed DesignState, PARAMETER REINSTATE, and graph schema v4 are hard prerequisites for Phases 5–8). Independent of the planned v4.0 BOT Bridge, but both touch `n8n/workflows/` — activation order between v4.0 and v9.0 is decided at milestone start via `/gsd-new-milestone`.

> **Isolation contract:** This document and `v9.0-REQUIREMENTS.md` are the only v9.0 artifacts. Nothing here modifies `.planning/phases/`, STATE.md, or the active v7.0 scope. Phase numbers below are milestone-local (1–8); global phase numbers are assigned when the milestone is activated.

## Overview

v9.0 upgrades the intelligence layer of the Design Grammar System along two axes:

1. **LLM infrastructure** — transition from local-only Ollama to a provider-agnostic **cloud LLM connector**: the user enters an API key in the UI and controls which external LLM (Anthropic Claude, OpenAI-compatible, or local Ollama fallback) manages Design Grammars. On top of the connector sits a **DG-aware context layer** that makes every LLM call ontology-aware: V7 concept catalog, SWRL violation-pattern conventions, and a standard Cypher expression catalog injected into prompts, with schema validation of generated Cypher before execution.

2. **AI implementation workflows** — the rules ingest/edit pipeline is rebuilt on the new connector (clarification loops, atom-level edit previews, validation-feedback retries); AI learns to **parse Grasshopper node blocks as parts of the Behaviour (`dg:Behavior`) of a Core Object (`dg:Object`)**, populating the ontology's Computgraph layer (`dgc:Algorithm`/`dgc:Procedure`/`dgc:Parameter`) at runtime for the first time; AI **generates candidate input parameter sets for Grasshopper scripts** as ParamState-compatible payloads; and an investigation phase prototypes **auto-validation of DesignState** (validation runs triggered by state capture, without a manual VALIDATOR trigger).

The milestone also contains a decision-gate phase evaluating a move of the orchestration layer from **n8n to OpenClaw** — spike, comparison, ADR; migration proceeds only on an explicit go decision, with n8n kept running in parallel until parity is proven.

**8 phases, estimated ~10–14 implementation sessions.**

## Phases

### Phase 1: Cloud LLM Connector and Provider Abstraction

**Goal:** Every LLM call in the system goes through one provider-agnostic gateway; the user controls provider, model, and API key from the UI, and the system falls back to local Ollama when no key is configured.

**Requires:** No prior v9.0 phases (foundation for everything else).

**Requirements:** LLMC-01, LLMC-02, LLMC-03, LLMC-04, LLMC-05, LLMC-06

**Deliverables:**
- `data-service/llm_gateway.py` — provider adapters (Anthropic Claude, OpenAI-compatible, Ollama) behind one `POST /llm/generate` contract: `{prompt, system, model?, provider?, options}` → `{text, provider, model, usage}`
- `data-service/app.py` — gateway endpoint + `GET/PUT/DELETE /llm/settings` (provider, model, masked key state) + `POST /llm/settings/test` (live key validation call)
- Encrypted-at-rest key storage (master secret via env var in `docker-compose.yml`; key never written to Neo4j, browser localStorage, or logs)
- `graph-viewer/index.html` — LLM settings panel: provider select, model select, API key input (masked), test button, active-provider indicator
- n8n workflows (`rules-to-metagraph.json`, `graph-query-mcp.json`) rewired: Ollama HTTP nodes replaced with gateway calls
- pytest coverage: adapter routing, fallback resolution, key masking, error mapping

**Success Criteria:**
1. With a valid Anthropic key saved in the UI, submitting a rule through the n8n webhook produces a graph written by Claude output; switching the provider dropdown to Ollama and resubmitting works without any restart
2. With no key configured, the system behaves exactly as today (Ollama) — regression-free
3. `grep -r "11435\|ollama" n8n/workflows/` shows no direct Ollama HTTP calls remaining (gateway only)
4. Quota/auth/timeout errors from a cloud provider surface in the UI with the What+Where+How-to-fix pattern; the API key never appears in any log or response payload

### Phase 2: DG-Aware Context Layer (SWRL + Ontology + Cypher Awareness)

**Goal:** LLM calls stop relying on a single static prompt — every ingest/query request is assembled from the V7 ontology catalog, SWRL violation-pattern conventions, and a standard Cypher expression catalog, and every generated Cypher statement is schema-validated before it touches Neo4j.

**Requires:** Phase 1 (context layer assembles prompts behind the gateway).

**Requirements:** CTXA-01, CTXA-02, CTXA-03, CTXA-04, CTXA-05

**Deliverables:**
- `data-service/dg_context.py` — deterministic context assembler: selects relevant ontology V7 concepts (per layer: Ontograph/Metagraph/Validgraph), SWRL atom patterns, and Cypher templates for the request type
- `llm/cypher_catalog.json` (new, versioned) — parameterized standard Cypher expressions for common rule shapes: max/min limit, range, ratio, boolean requirement, existence/count — each with a worked SWRL + Cypher example
- SWRL convention block in machine-readable form (violation-inverted body semantics, atom ordering, Var/Literal argument rules)
- `data-service` Cypher validator — checks generated Cypher against the current schema template (labels, relationship types, `kind` enums, property names, bracket nesting) and returns structured violations for the retry loop
- n8n prompt nodes reduced to thin callers of the context assembler

**Success Criteria:**
1. The ingest prompt for a "maximum height" rule demonstrably contains the max-limit Cypher template and the V7 concepts it references (inspectable via a debug endpoint)
2. Deliberately corrupting the LLM output (wrong label, wrong kind value) is caught by the validator before execution and returned as a structured violation list
3. Context selection is fully deterministic — same request, same context, no embeddings involved
4. A natural-language graph query about design states answers correctly using v4 `kind` values with the new context layer active

### Phase 3: Orchestration Evaluation — n8n vs OpenClaw

**Goal:** A grounded go/no-go decision on replacing n8n with OpenClaw as the orchestration layer, backed by a working spike — not opinion.

**Requires:** Phase 1 (any orchestrator must call the LLM gateway; evaluating pre-gateway would measure the wrong architecture). Can run parallel to Phase 2.

**Requirements:** ORCH-01, ORCH-02, ORCH-03, ORCH-04

**Deliverables:**
- Evaluation matrix: webhook parity (n8n's 2 DG webhooks), function-node equivalents, LLM provider support, self-host Docker footprint, state/retry handling, maintenance and community health, migration cost — n8n vs OpenClaw
- Spike: the `graph-query` workflow (simpler of the two) ported to OpenClaw against the live stack, exercising the same `/n8n/webhook/dg/graph-query` contract behind the Nginx proxy
- ADR in `DG_OBSIDIAN/knowledge/decisions/` recording the decision with explicit criteria; a no-go leaves n8n untouched and closes the question
- If go: phased migration plan — one workflow at a time, n8n kept running in parallel as fallback until parity is verified end-to-end

**Success Criteria:**
1. The spike answers with evidence: can OpenClaw serve the DG webhook workflows (request → LLM gateway → Neo4j → response) with equivalent latency and error handling?
2. The ADR states go/no-go, the criteria that decided it, and — on go — which milestone executes the migration
3. No production workflow is switched in this phase; n8n remains the active orchestrator throughout v9.0 unless the ADR explicitly schedules cut-over inside Phase 4

### Phase 4: Rules Ingestion and Editing Workflow Upgrade

**Goal:** The rules ingest and edit workflows exploit the cloud LLM and context layer: ambiguity triggers clarification instead of guessing, edits preview atom-level changes before committing, and failed generations self-correct through validator feedback.

**Requires:** Phases 1–2; Phase 3 ADR (determines whether the upgraded workflows are built in n8n or OpenClaw).

**Requirements:** RING-01, RING-02, RING-03, RING-04, RING-05

**Deliverables:**
- Upgraded ingest workflow: gateway + context assembler + validator-feedback retry loop (bounded attempts, then actionable error)
- Clarification loop: ambiguous rules (missing limit value, unknown concept, ambiguous unit) return a clarification question to the UI instead of a guessed graph
- Edit flow with preview: proposed atom changes rendered as old→new diff in the UI before the MATCH-DELETE + re-create commit
- Regression suite: reference rule set from `test/` fixtures run through both cloud and Ollama paths

**Success Criteria:**
1. The reference rule set ingests with a pass-rate ≥ the current Ollama baseline (measured, not assumed), and cloud-path results validate against the schema template with zero manual fixes
2. Submitting "the building must not be too tall" yields a clarification question (which limit?), not a fabricated rule
3. Editing a rule shows the atom diff; confirming applies it with old-atom cleanup intact; cancelling leaves the graph unchanged
4. Disabling the API key mid-session degrades gracefully to Ollama with a visible provider indicator — no broken workflow state

### Phase 5: Grasshopper Node Recognition as Object Behaviour

**Goal:** AI reads a Grasshopper definition and recognizes blocks of nodes as parts of the Behaviour (`dg:Behavior`) of a Core Object (`dg:Object`) — populating the ontology's Computgraph layer at runtime for the first time.

**Requires:** Phases 1–2 (recognition prompts need the gateway and the ontology-aware context layer). v7.0 GH components shipped (milestone prerequisite).

**Requirements:** GHRC-01, GHRC-02, GHRC-03, GHRC-04, GHRC-05

**Deliverables:**
- GH definition export path: serializer of the canvas (or GHX/JSON upload) into a node/wire/group structure suitable for LLM analysis — component names, parameter names, group labels, wiring topology
- Recognition pipeline: AI segments the definition into node blocks and classifies each block's role within the Object's Behaviour (per the FBS framing: Behavior IS the parametric definition that generates design output)
- Persistence to the Computgraph layer: `dgc:Algorithm` (whole definition) → `dgc:Procedure` (node blocks) → `dgc:Parameter` (inputs), linked to the Core Object via `dg:hasBehavior` and the Behavior→Algorithm relation — project-isolated, MERGE-idempotent
- NeoVis Computgraph view: `config.template.js` styling + graph-viewer toggle for the new layer
- Schema propagation for the new labels per the standard checklist (`cypher_template.txt`, `dataset_schema.json`, n8n/orchestrator prompts, NeoVis config)

**Success Criteria:**
1. Submitting a real Frame-generating GH definition produces a Computgraph subgraph: one Algorithm, ≥2 Procedures matching the definition's logical blocks, Parameters matching its sliders — linked to the Frame Object via `hasBehavior`
2. Re-submitting the same definition creates no duplicate nodes (MERGE idempotency verified by count query)
3. The graph viewer renders the Computgraph layer distinctly, filterable per project
4. A definition with unrecognizable blocks degrades gracefully: recognized blocks persist, unrecognized ones are reported, nothing is invented

### Phase 6: AI-Generated Grasshopper Script Inputs

**Goal:** Given a rule and the recognized Behaviour structure, AI proposes concrete input parameter sets for the Grasshopper script — delivered as ParamState-compatible payloads the architect can review and apply via PARAMETER REINSTATE.

**Requires:** Phase 5 (Parameter structure comes from recognition); v7.0 PARAMETER REINSTATE (milestone prerequisite).

**Requirements:** GHIN-01, GHIN-02, GHIN-03, GHIN-04

**Deliverables:**
- Input-generation endpoint: rule (or design intent text) + recognized `dgc:Parameter` set → N candidate parameter assignments, typed Number/Integer/Boolean only (standing v2.0 decision)
- Candidates serialized as ParamState-compatible payloads (statePayloadJson v2) with provenance properties: source rule, provider/model, timestamp
- UI review surface: candidate list with per-parameter values, accept/reject; accepted candidates stored as ParamStates
- Application path: accepted ParamState applied on the canvas through the existing PARAMETER REINSTATE component — no new apply mechanism

**Success Criteria:**
1. For a max-height rule against the Frame definition, generated candidates keep every parameter inside its recognized slider domain and satisfy the rule's limit where determinable
2. An accepted candidate round-trips: stored as ParamState → visible in VALIDATION GRAPH reads → applied via PARAMETER REINSTATE with per-parameter ReStatus reporting
3. Nothing touches the canvas without explicit user acceptance — generation and application are strictly separated
4. Provenance is queryable: `MATCH` on generated ParamStates returns rule, model, and timestamp for each

### Phase 7: DesignState Auto-Validation Investigation

**Goal:** A prototyped, evidence-based answer to "can validation run automatically when a DesignState is captured?" — architecture chosen, guardrails defined, full implementation scoped for a follow-up milestone.

**Requires:** v7.0 VALIDATOR + composed DesignState shipped (milestone prerequisite). Independent of Phases 3–6; can run parallel.

**Requirements:** DSAV-01, DSAV-02, DSAV-03

**Deliverables:**
- Investigation note comparing trigger architectures: (a) capture-time hook in the DESIGN STATE / VALIDATOR components (GH-side), (b) data-service watcher on DesignState writes, (c) event-driven trigger on Neo4j ValidGraph writes — with latency, publish-flood, and Speckle-noise analysis for each
- Working prototype of at least one path: capturing a new DesignState produces a validation Run without a manual VALIDATOR trigger
- Guardrail design: debounce window, per-project rate limit, opt-in flag (auto-validation must be explicitly enabled per project)
- ADR in `DG_OBSIDIAN/knowledge/decisions/` scoping full implementation to a future milestone

**Success Criteria:**
1. The prototype demonstrably closes the loop on live Docker: DesignState captured → Run appears in ValidGraph and (if enabled) publishes to Speckle — hands-off
2. Rapid successive captures do not flood Speckle: the debounce/rate-limit design is validated in the prototype, not just described
3. The ADR records the chosen architecture, the rejected options with reasons, and the follow-up milestone scope

### Phase 8: E2E Validation and Docs

**Goal:** The full v9.0 intelligence chain runs end-to-end on live Docker under a cloud provider, provider switching is friction-free, and every new service, setting, and workflow is documented.

**Requires:** Phases 1–7.

**Requirements:** INTG-01, INTG-02, INTG-03, INTG-04

**Deliverables:**
- E2E run: NL rule → cloud-LLM ingest (context layer + validation) → graph → GH validation → Speckle publish, on live Docker
- E2E GH intelligence chain: GH definition parsed → Behaviour in Computgraph → generated inputs accepted → applied via PARAMETER REINSTATE → validation run (auto or manual per Phase 7 outcome)
- Provider-switch drill: Claude ↔ OpenAI-compatible ↔ Ollama via settings only, mid-session, no restarts
- Docs: CLAUDE.md service map + gotchas, `spec/`, DG_OBSIDIAN notes (decisions from Phases 3 and 7 filed), graphify refresh

**Success Criteria:**
1. Both E2E chains complete without errors in one session on live Docker
2. Provider switching requires only the settings panel — verified for all three providers
3. All 36 v9.0 requirements are checked off in v9.0-REQUIREMENTS.md with traceability complete
4. `grep -ri "ollama" CLAUDE.md spec/` presents Ollama as the fallback provider, not the sole LLM path
