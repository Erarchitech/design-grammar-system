# Requirements: Design Grammar System v4.0 â€” BOT Ontology Bridge

**Defined:** 2026-05-25
**Status:** Planned (pending v3.0 completion)
**Core Value:** When architects write rules mentioning spatial concepts (building, storey, space, site), the DG system automatically aligns the generated ontology entities to the W3C Building Topology Ontology (BOT), making the Ontograph interoperable with external BIM standards without any manual work from the user.

## v4.0 Requirements

### BOT Concept Matching

- [ ] **BOTM-01**: The system maintains an in-memory BOT vocabulary reference covering the 6 BOT classes (Building, Storey, Space, Site, Zone, Element) and 6 BOT object properties (hasBuilding, hasStorey, hasSpace, containsZone, adjacentZone, hasElement), including common English synonyms (floorâ†’Storey, roomâ†’Space, plotâ†’Site, etc.)
- [ ] **BOTM-02**: Matching is case-insensitive, synonym-aware, and operates on the Class/ObjectProperty label extracted from user-submitted rules â€” no LLM involvement in the matching itself
- [ ] **BOTM-03**: A `GET /ontograph/bot-mappings` endpoint exposes the full mapping table as JSON for use by n8n and the UI
- [ ] **BOTM-04**: A `POST /ontograph/align-bot` endpoint accepts `{project, ex_iri, label, node_type}`, performs the match, and â€” if matched â€” creates the BOT anchor node and `ALIGNED_TO` edge in Neo4j; returns `{"aligned": false}` with no DB writes if unmatched

### On-Demand BOT Node Creation

- [ ] **BOTO-01**: BOT anchor nodes (`BOTClass`, `BOTProp`) are created in Neo4j **only when a rule ingest triggers a match** â€” no bulk pre-seeding of the full BOT vocabulary
- [ ] **BOTO-02**: Each BOT anchor node has properties `iri` (e.g., `bot:Building`), `label`, `botNamespace='bot'`, `graph='BOTVocab'`
- [ ] **BOTO-03**: The `ALIGNED_TO` relationship connects an `ex:` OntoGraph node to its `BOTVocab` anchor: `(:Class {iri:'ex:Building'})-[:ALIGNED_TO]->(:BOTClass {iri:'bot:Building'})`
- [ ] **BOTO-04**: MERGE idempotency is preserved â€” ingesting a second rule with the same BOT concept reuses the existing BOT anchor node and `ALIGNED_TO` edge; no duplicates created
- [ ] **BOTO-05**: Custom `ex:` classes that have no BOT match are created as normal with no `ALIGNED_TO` edge â€” existing behaviour is unchanged for unmatched concepts

### n8n Workflow Enhancement

- [ ] **BOTN-01**: A "Fetch BOT Context" Function node is added to `rules-to-metagraph.json` before the Build LLM Prompt node; it retrieves the BOT mapping table and makes it available for prompt injection
- [ ] **BOTN-02**: The LLM system prompt includes a BOT alignment hint section listing the mapping table and instructing the LLM to emit `ALIGNED_TO` blocks for matched classes
- [ ] **BOTN-03**: An "Auto-Align BOT" Function node is added after the Execute LLM Cypher node; it parses created Class/ObjectProperty labels from the executed Cypher, calls `POST /ontograph/align-bot` for each, and creates any `ALIGNED_TO` edges the LLM may have omitted
- [ ] **BOTN-04**: The "Auto-Align BOT" node is a deterministic safety net â€” alignment correctness does not depend on the LLM following the hint

### Schema Propagation

- [ ] **BOTS-01**: `cypher_template.txt` lists `BOTClass` and `BOTProp` as allowed node labels and `ALIGNED_TO` as an allowed relationship type, with a note that these are only emitted on BOT matches
- [ ] **BOTS-02**: `training/dataset_schema.json` contains a `BOTVocab` graph layer entry documenting on-demand creation semantics, namespace prefix `bot`, and node labels `BOTClass`/`BOTProp`
- [ ] **BOTS-03**: NeoVis config (`graph-viewer/config.template.js`) includes distinct styling for `BOTClass` nodes (color `#4A90D9`) and dashed `ALIGNED_TO` edges so BOT alignment is immediately visible in the graph viewer

### Integration

- [ ] **INTG-01**: End-to-end flow: user submits "Maximum building height is 75m" â†’ rule ingested â†’ `ex:Building` node has `ALIGNED_TO` edge to `bot:Building` node in `BOTVocab` graph â€” confirmed by Cypher query
- [ ] **INTG-02**: Synonym resolution works: "each storey must be at least 2.8m" maps `ex:Storey` to `bot:Storey`; "apartment floor area" maps `ex:Apartment` to `bot:Space`
- [ ] **INTG-03**: No-match case is clean: a rule with a non-BOT concept (e.g., `ex:FacadePanel`) creates no BOT node, no `ALIGNED_TO` edge, and the rule evaluates correctly via the existing C# validation path
- [ ] **INTG-04**: Re-ingesting a rule with the same BOT concept does not create duplicate BOT nodes (MERGE idempotency verified by node count query)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Pre-seeding all 20+ BOT entities on startup | Keeps Neo4j lean; BOT grows only as rules reference concepts |
| IFC/Industry Foundation Classes alignment | Separate vocabulary; deferred to a future milestone |
| Embedding-based fuzzy concept matching | Synonym dictionary covers the common surface forms; embeddings add infra complexity without clear gain at this stage |
| Replacing `ex:` prefix with `bot:` IRIs | Would break the C# parser, LLM prompt conventions, and existing Cypher templates |
| C# plugin surfacing `BotIri` in Grasshopper | Display-only enrichment; deferred to Phase 4 as optional |
| Full OWL reasoning over BOT hierarchy | Not required; ALIGNED_TO is a flat assertion link, not a subsumption axiom |

---

## Key Decisions (v4.0)

| Decision | Rationale |
|----------|-----------|
| `ALIGNED_TO` edge, not prefix replacement | Preserves all existing `ex:` conventions across C#, LLM, and Cypher; BOT is additive metadata |
| In-memory mapper, not a DB vocabulary table | BOT vocabulary is stable (W3C spec); no need to round-trip through Neo4j for matching |
| Post-processing safety net in n8n | LLM hint is best-effort; deterministic Python alignment guarantees correctness regardless of LLM output |
| `BOTVocab` graph tag, not `OntoGraph` | Keeps BOT anchor nodes visually and queryably separate from project-specific `ex:` ontology terms |
| On-demand node creation | Only the concepts actually used by a project appear in its graph â€” no vocabulary pollution |
