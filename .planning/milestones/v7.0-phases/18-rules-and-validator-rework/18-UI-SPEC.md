## Phase Scope: Only GHVL-06

The frontend adaptation for Phase 18 is limited to GHVL-06: **"The Model Viewer keeps working against v7.0 data."** No new UI is being built. The changes are:

### 1. Data-service `_project_state_summary` — v2 envelope detection (REQUIRED)

**Current behavior (lines 534-558 of `data-service/app.py`):**
- Parses `statePayloadJson` as JSON
- Reads `stateId`, `capturedAtUtc`, `parameters` from parsed dict
- Returns `{stateId, capturedAtUtc, parameterCount}` — never raises on malformed input

**Required change:**
- Check `parsed.get("version")` to detect v2 envelope
- If `version == "2"`: count elements from `objStates`, `paramStates`, `propStates` arrays instead of `parameters`
- If v1 fallback (no version field or version != "2"): existing behavior unchanged
- Return shape remains identical: `{stateId, capturedAtUtc, parameterCount}`

**Contract:**
```python
# v2 input:
# {"version":"2","stateId":"DS_abc","capturedAtUtc":"...",
#  "objStates":[...], "paramStates":[...], "propStates":[...]}
# Returns: {"stateId":"DS_abc", "capturedAtUtc":"...", "parameterCount": N}

# v1 input (unchanged):
# {"stateId":"S-001","capturedAtUtc":"...","parameters":[...]}
# Returns: {"stateId":"S-001", "capturedAtUtc":"...", "parameterCount": N}
```

**The return shape is the same for both v1 and v2.** The frontend grouping code (`useValidationRunsGrouping.js`) already reads `run.state.stateId` — no change needed on the JS side for basic operation.

### 2. `useValidationRunsGrouping.js` — no change required, but label enhancement suggested (OPTIONAL)

**Current behavior:**
- `deriveStateKey(run)` reads `run.state.stateId` and uses it as both group key and group label
- Group-by-State mode groups runs by `stateId`

**Why no change is required:** The `_project_state_summary` function for v2 payloads will still return `stateId`, `capturedAtUtc`, and `parameterCount`. The grouping function derives its key from `run.state.stateId` — this field path is preserved across v1 and v2 projections.

**Suggested label enhancement (designer discretion):** For v2 payloads, the group label could be enriched to show the composition, e.g. "DS_abc (3 obj, 5 param)" instead of just "DS_abc". This requires adding a `parameterCount` → `groupLabel` derivation in `deriveStateKey`. The `_project_state_summary` already returns `parameterCount` — but for v2, this count is the sum of objStates + paramStates + propStates. Label format (if implemented): `{stateId}` for v1, `{stateId} ({N} total)` for v2.

### 3. `ValidationRunsStrip.jsx` — no change required (OPTIONAL)

**Current behavior:** Renders group label from `group.groupLabel`. In state mode this is the `stateId`. No change needed for basic v2 compatibility.

**Optional enhancement (if label enrichment from #2 is implemented):** The render path is already generic — it displays whatever `groupLabel` contains. No JSX changes needed.

## Data Contract (Read-Side Only)

### API response shape (`/validation/runs/{project}`)

The `state` field within each run object is the output of `_project_state_summary`. For **both v1 and v2**, the shape is:

```json
{
  "runId": "...",
  "state": {
    "stateId": "DS_abc | S-001",
    "capturedAtUtc": "2026-...",
    "parameterCount": 5
  },
  ...
}
```

### Consumption by `useValidationRunsGrouping.js`

The grouping function accesses:
- `run.state.stateId` → group key and label
- `run.createdAt` → sort order (DESC)
- `run.runId` → tiebreaker sort

When `state` is `null`, the run goes into the `"__no_state__"` bucket. When `stateId` is empty string, same fallback. These behaviors are unchanged.

### Version detection strategy

- **v2 detection:** Check `parsed.get("version") == "2"` at the root of statePayloadJson
- **v1 fallback:** No `version` field present, or `version != "2"` — read `parameters` array as before
- **Safety:** Never raise on malformed input. Return `None` for unparseable JSON, empty/missing payload
