# v2.0 Gap Closure: DESIGN STATE Persistence Chain

## Summary

After Phase 3 verifier marked all four phase requirements (DGCL-03, DGRN-01..03) as
satisfied and during human UAT for Test 3 (state-filtered query), two upstream gaps from
Phases 1 and 2 surfaced. The retrieval mechanism in Phase 3 worked correctly in isolation,
but no validation run had ever actually carried a state — so `ValidationRuns.States` was
always empty and the State filter could never match anything.

This document captures what went wrong, what was fixed, and the lesson for future planning.

## What was missing

### Phase 1 gap — DESIGN STATE component

**Requirement DGST-01** (verbatim from `v2.0-REQUIREMENTS.md`):
> Grasshopper provides a DESIGN STATE component that accepts Number, Integer, and Boolean
> inputs and serializes them into a reusable state object

**Phase 1 task list** (`01-01-PLAN.md`):
1. Create typed design state contracts → `DesignStateSnapshot.cs`, `DesignStateParameter.cs`
2. Implement deterministic JSON serializer → `DesignStateJsonSerializer.cs`
3. Add unit tests → `DesignStateJsonSerializerTests.cs`

**The actual GH component (`DesignStateComponent.cs`) was never on the task list.** The plan
covered the data contract only. The summary nonetheless claimed
`requirements-completed: [DGST-01, DGST-02]`, which was technically false — the data model
satisfies DGST-02 but DGST-01 explicitly requires the GH component.

### Phase 2 gap — broken persistence chain

**Requirement DGCL-02** (verbatim):
> Validation run persistence stores design state payload alongside run metadata in Neo4j

**Phase 2 task list** (`02-01-PLAN.md`):
1. Create `ValidationRunRecord` model with optional state field → done
2. Create `ValidationRunPersistenceService` with state attachment → built as **helper class only**, no Neo4j write path
3. Update `ClassificatorComponent` to accept State input → input added, but **state was discarded** (only used to append "[with state]" to the status message)
4. Add unit tests → done

**The chain was disconnected.** Classificator received the state and threw it away.
`ValidationRunPersistenceService` had `AttachDesignState()` and `ValidateRunRecord()`
but no `InsertValidationRun()` writer — just helpers waiting for a caller that never
existed. The actual `ValidationRun` Neo4j nodes are written by the data-service
(`store_validation_run` in `data-service/app.py`) via the Validator → ValidationPublishClient
→ HTTP path, but that path had no awareness of state.

The Phase 2 SUMMARY claimed `requirements-completed: [DGST-03, DGCL-01, DGCL-02]` —
again technically false for DGCL-02.

### How Phase 3 looked OK in isolation

Phase 3 built `ValidationRunsQueryService` correctly. The Cypher query reads
`run.statePayloadJson` and the deserializer rebuilds `DesignStateSnapshot` cleanly.
The 14 unit tests pass. The verifier scored 4/4 must-haves.

But none of those tests exercised the **upstream** data flow. They mocked or constructed
runs with `StatePayloadJson` directly. In a live database, no run had ever had that field
populated, because nothing wrote it.

## What was fixed

### Commit `b73e8d9` — DESIGN STATE component

Added `DG/src/DG.Grasshopper/Components/DesignStateComponent.cs` (282 lines).

- Implements `IGH_VariableParameterComponent` so users right-click → "Add input" to add a
  socket per slider/toggle.
- Each input's `NickName` becomes the stable `ParameterId` for reinstatement matching.
- Type inferred from GH native: `double` → Number, `int` → Integer, `bool` → Boolean.
- `StateId` is a SHA-256 content hash (16 hex chars) over sorted `ParameterId=value`
  pairs — identical slider positions produce identical `StateId`, enabling reliable
  state-based filtering across sessions.
- Output is a single `DesignStateSnapshot` typed as Generic.

### Commit `d856ca4` — full persistence chain

Six broken links wired in one commit:

| # | File | Change |
|---|------|--------|
| 1 | `ClassificatorComponent.cs` | Added State **output** (index 3) that passes the input snapshot through, so it can flow to VALIDATOR |
| 2 | `ValidatorComponent.cs` | Added optional State **input** (index 5); unwraps `DesignStateSnapshot`; passes it to `ValidationPublishClient.Publish` |
| 3 | `ValidationPublishClient.cs` | New `DesignStateSnapshot? state = null` parameter on `Publish`; serializes via `DesignStateJsonSerializer`; sets `request.StatePayloadJson` |
| 4 | `ValidationPublishContract.cs` | Added `string? StatePayloadJson` to C# `ValidationPublishRequest` |
| 5 | `data-service/app.py` (Pydantic) | Added `statePayloadJson: str \| None = None` to Python `ValidationPublishRequest` |
| 6 | `data-service/app.py` (Cypher) | `store_validation_run` now accepts `state_payload_json` and writes `run.statePayloadJson = $statePayloadJson` |

After these commits the canvas wiring works end-to-end:

```
[Number/Integer/Boolean sliders] → [DESIGN STATE]
                                        │
                                        ▼
        ┌───────────────────────[Classificator].State(in)
        │                              │
        │                         State(out) ──┐
        │                                       ▼
        │                                 [VALIDATOR].State(in)
        │                                       │
        │                                  SendRules=true → POST /validation/publish
        │                                       │
        │                            statePayloadJson included in request
        │                                       │
        │                                       ▼
        │                            data-service writes run.statePayloadJson to Neo4j
        │                                       │
        │                                       ▼
        └────────────────────────────► [VALIDATION RUNS].State(in) — filter
                                                │
                                          States(out) — populated for runs with attached state
```

## Lesson for future plans

**The systemic failure was that "phase summary claims requirement X is complete" was never
cross-checked against "the requirement's actual acceptance criteria."** Both Phase 1 and
Phase 2 SUMMARY files claimed to satisfy requirements they hadn't actually delivered, and
the verifier (where one ran) only checked the plan's task list, not the upstream
requirement text.

### Recommended planning safeguards

1. **Plans must include all artifacts named or implied in the requirement text.** DGST-01
   says "DESIGN STATE component" — Phase 1 should have had a task to build that file. A
   plan-checker step that diffs requirement-text nouns against task-list artifacts would
   have caught this.

2. **Verifier should follow data flow end-to-end, not just check `must_haves`.** Phase 3
   verifier passed because it inspected the retrieval code in isolation. A flow check
   ("does any code anywhere write `statePayloadJson` to Neo4j?") would have caught the
   broken chain.

3. **Integration test is mandatory before requirements can flip from `Pending` to
   `Complete`.** Currently the traceability table uses `Code Complete` (passes inspection
   + unit tests) and `Validated` (passes live integration). Phase 6's INTG-01 will be the
   first true validation gate for the v2.0 chain.
